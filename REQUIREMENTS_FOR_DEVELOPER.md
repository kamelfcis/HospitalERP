## Hospital General Ledger System — نظام محاسبة المستشفيات

**تاريخ الإعداد:** أبريل 2026  
**المُعِد:** مراجعة فنية شاملة من المطور الأصلي للنظام

## أولاً — الواقع بصدق

النظام **مكتوب ويعمل** لكن **لم يخضع لاختبار حقيقي منهجي** على أي بيئة إنتاج فعلية.  
كل ما هو موجود تم اختباره أثناء التطوير فقط (developer testing)، وليس user acceptance testing.

> **التقدير الحقيقي للجاهزية: 72% — ليس 90%**  
> الفرق هو أن الـ 28% الباقية هي أخطاء لم تُكتشف بعد + ميزات ناقصة.

### 2.1 مسارات يجب اختبارها يدوياً أولاً

#### مسار (أ) — دورة المشتريات الكاملة
```
مورد جديد
    ↓
أمر استلام بضاعة (بدفعات وتواريخ انتهاء مختلفة)
    ↓
فاتورة شراء مرتبطة بالاستلام
    ↓
دفع للمورد (جزئي ثم كامل)
    ↓
مردود مشتريات جزئي
    ↓
التأكد: رصيد المورد صحيح، المخزون صحيح، القيد المحاسبي تولّد
```
**الأخطاء المتوقعة:**
- القيد المحاسبي لا يتولّد إذا الحساب غير مربوط في Account Mappings
- مردود المشتريات قد لا يُعيد الدفعة (lot) الصحيحة للمخزون
- الأرقام التسلسلية قد تتكرر في بيئة concurrent

---

#### مسار (ب) — دورة المبيعات الكاملة (صيدلية)
```
فتح وردية كاشير
    ↓
فاتورة بيع بباركود (FEFO تلقائي)
    ↓
فاتورة بيع لعميل آجل (عقد)
    ↓
مردود مبيعات جزئي
    ↓
تحصيل من عميل آجل
    ↓
إغلاق الوردية وتسليم
    ↓
التأكد: المخزون صحيح، الكاشير يوافق، القيد تولّد
```
**الأخطاء المتوقعة:**
- الـ FEFO قد يختار دفعة منتهية الصلاحية بدلاً من الأقرب للانتهاء
- تسليم الكاشير قد لا يحسب المبالغ المختلطة (نقدي + آجل) بشكل صحيح
- SSE قد ينقطع في الشبكات البطيئة

---

#### مسار (ج) — دورة المريض الكاملة (IPD)
```
تسجيل مريض جديد
    ↓
إدخال (Admission) وتخصيص سرير
    ↓
فتح فاتورة إقامة
    ↓
إضافة خدمات + أدوية + إقامة
    ↓
دفع جزئي من الكاشير
    ↓
تحويل نصيب الطبيب
    ↓
إقفال الفاتورة (Finalize)
    ↓
إغلاق الإقامة وتحرير السرير
    ↓
Final Close
    ↓
التأكد: قيد GL تولّد، رصيد المريض صفر، السرير متاح
```
**الأخطاء المتوقعة:**
- قيد GL لا يتولّد لـ 32 فاتورة موجودة حالياً (مؤكد)
- Stay Engine قد يحسب أيام الإقامة خطأ عند تغيير السرير
- إقفال الفاتورة بمريض عقد قد يفشل إذا الشركة بدون حساب ذمم

---

#### مسار (د) — دورة المريض الخارجي (OPD)
```
حجز موعد في العيادة
    ↓
استقبال المريض وقياسات
    ↓
الكشف الطبي (تشخيص + روشتة)
    ↓
أوامر للمعمل/الأشعة
    ↓
صرف دواء من الصيدلية
    ↓
تحصيل الكاشير
    ↓
التأكد: القيد المحاسبي (IFRS deferral) صحيح
```
**الأخطاء المتوقعة:**
- أوامر المعمل تُنشأ لكن النتائج لا ترجع للملف الطبي (غائب كلياً)
- قيد دفع مقدم ثم قيد إيراد قد يختل في بعض السيناريوهات

---

#### مسار (هـ) — دورة العقود والتأمين
```
إنشاء عقد تأمين مع قواعد تغطية
    ↓
تسجيل مريض مرتبط بالعقد
    ↓
فاتورة مريض عقد مع تطبيق نسب التغطية
    ↓
موافقة مسبقة (Approval)
    ↓
إنشاء دفعة مطالبة (Claim Batch)
    ↓
تسوية جزئية من شركة التأمين
    ↓
التأكد: ذمم الشركة صحيحة، نصيب المريض صحيح
```
**الأخطاء المتوقعة:**
- قواعد التغطية المعقدة (سقف + نسبة + استثناءات) قد تُطبَّق خطأ
- تسوية جزئية قد لا تُوزَّع صح على البنود
- هذا المسار لم يُختبر مع بيانات تأمين حقيقية مطلقاً

---

#### مسار (و) — التحويلات والجرد
```
تحويل أصناف من مخزن لمخزن
    ↓
جرد دوري وتسجيل فروقات
    ↓
ترحيل الجرد وتعديل المخزون
    ↓
التأكد: الأرصدة صحيحة في كل مخزن
```
**الأخطاء المتوقعة:**
- فروقات الجرد قد لا تُولّد قيد محاسبي صحيح
- التحويل المعلق (pending) قد يحجز كميات ويمنع البيع

### 2.2 اختبارات تقنية مطلوبة (Automated Tests)

هذه الاختبارات **غير موجودة** في المشروع حالياً ويجب كتابتها:

#### أ) اختبارات API (Backend Integration Tests)
```
الأدوات المقترحة: Vitest + Supertest
الأولوية: عالية جداً

المسارات الأهم للتغطية:
□ POST /api/patient-invoices → finalize → final-close (كامل)
□ POST /api/sales-invoices → finalize (مع lot deduction)
□ POST /api/receiving → post (مع GL generation)
□ POST /api/cashier/open-shift → collect → close-shift
□ POST /api/cash-transfers (مع GL journal)
□ PATCH /api/patient-invoice-payments (دفع مريض)
□ التأكد من أن checkPermission يرفض بدون صلاحية (401/403)
```

#### ب) اختبارات قاعدة البيانات
```
□ Race condition في admission_number_seq (concurrent requests)
□ FEFO يختار الدفعة الأقرب انتهاءً دائماً
□ Oversell guard يمنع البيع بالسالب فعلاً
□ Row locks (FOR UPDATE) تمنع التعديل المتزامن
```

#### ج) اختبارات الواجهة (E2E)
```
الأدوات المقترحة: Playwright
المسارات الأهم:
□ تسجيل دخول + تحقق صلاحيات
□ إنشاء فاتورة مريض كاملة
□ إغلاق وردية كاشير
□ طباعة إيصال (تحقق A4 و thermal)
```

## رابعاً — الميزات الناقصة (مرتبة بالأولوية)

### 4.1 أولوية عالية — تؤثر على التشغيل اليومي

#### أ) تقارير تشغيلية مفقودة
الجداول موجودة وتُملأ تلقائياً لكن لا توجد شاشات تعرضها:
```
□ تقرير الإيرادات اليومية (rpt_daily_revenue)
   - الإيراد بالقسم والصيدلية يومياً
   - مقارنة نقدي vs آجل vs تأمين

□ تقرير نشاط الأقسام (rpt_department_activity)
   - نسبة إشغال الأسرة يومياً
   - عدد المدخلين والخارجين

□ لوحة تحكم مالية للإدارة (Dashboard)
   - الإيراد اليومي، الأسبوعي، الشهري
   - أعمار الذمم (30/60/90 يوم)
   - تنبيهات: مخزون منخفض، قيود فاشلة، فواتير معلقة
```

#### ب) إدارة المرتجعات المالية
```
□ سيناريو: مريض دفع وأُلغيت الفاتورة — كيف يُرد المبلغ؟
   الحالي: لا توجد آلية رد مبالغ للمريض
□ سيناريو: كاشير أخطأ في مبلغ التحصيل
   الحالي: لا يوجد تعديل بعد الإغلاق
```

#### ج) إشعارات داخلية
```
□ تنبيه: فاتورة معلقة أكثر من X أيام
□ تنبيه: مخزون أقل من الحد الأدنى
□ تنبيه: قيد محاسبي فشل
□ تنبيه: عقد تأمين على وشك الانتهاء
```

### 4.2 أولوية متوسطة — تكتمل الدورة الطبية

□ تاريخ عمليات سابقة
□ ملاحظات التمريض
```

#### ج) نظام المواعيد المتكامل
```
الحالي: حجز موعد بسيط
الناقص:
□ تأكيد موعد عبر SMS/WhatsApp
□ تذكير تلقائي قبل الموعد
□ قائمة انتظار تلقائية عند الإلغاء

## خامساً — متطلبات البنية التحتية

### 5.1 مطلوب قبل Deployment للإنتاج
```
□ CI/CD Pipeline (GitHub Actions مثلاً)
   - تشغيل الاختبارات عند كل commit
   - منع deploy إذا الاختبارات فشلت

□ نسخ احتياطي تلقائي (Automated Backup)
   - pg_dump يومياً تلقائي
   - حفظ في مكان خارجي (S3 أو ما يعادله)
   - اختبار استعادة كل أسبوع

□ Monitoring & Alerts
   - تنبيه إذا الخادم توقف
   - تنبيه إذا الـ DB وصلت لحد معين
   - تنبيه إذا error rate ارتفع

□ Environment Separation
   - بيئة Development (للتطوير)
   - بيئة Staging (للاختبار)
   - بيئة Production (للإنتاج)
   - حالياً كل شيء في بيئة واحدة
```

### 5.2 أمان (Security)
```
□ Rate Limiting على API Login (منع brute force)
   حالياً: لا يوجد حد لمحاولات الدخول

□ HTTPS إجباري في الإنتاج
□ Session timeout (تسجيل خروج تلقائي بعد خمول)
   حالياً: الجلسة لا تنتهي

□ تشفير البيانات الحساسة في DB
   (كلمات السر مشفرة، لكن بيانات المرضى غير مشفرة)

□ Audit trail شامل
   حالياً: موجود جزئياً — بعض العمليات لا تُسجَّل
```

## سابعاً — نقاط يجب يعرفها المبرمج عن الكود

### أشياء تبدو منطقية لكنها خطأ في هذا المشروع:
```typescript
// ❌ خطأ — لا يوجد req.user
const userId = req.user?.id;

// ✅ الصحيح
const userId = req.session.userId;

// ❌ خطأ — query param كـ array element ثاني
useQuery({ queryKey: ["/api/items", { deptId }] })

// ✅ الصحيح
useQuery({ queryKey: ["/api/items"], queryFn: () => apiRequest(`/api/items?deptId=${id}`) })

// ❌ خطأ — SelectItem بدون value
<SelectItem value="">الكل</SelectItem>

// ✅ الصحيح
<SelectItem value="__all__">الكل</SelectItem>

// ❌ خطأ — roundMoney يعيد string
const total = roundMoney(a) + roundMoney(b);

// ✅ الصحيح
const total = parseFloat(roundMoney(a)) + parseFloat(roundMoney(b));
```

### البيانات الحساسة في الكود:
```
- اسم عمود الأقسام: name_ar (وليس name)
- اسم عمود المرضى: full_name (وليس name)
- المستخدم الافتراضي: admin / admin123 (يجب تغييره في الإنتاج)
```
