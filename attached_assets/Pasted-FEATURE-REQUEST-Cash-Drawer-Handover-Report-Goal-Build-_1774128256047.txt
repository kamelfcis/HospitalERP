BUSINESS DEFINITIONS (MUST BE EXPLICIT)
==================================================

You must inspect the current schema and existing business logic in the codebase.
Do not invent definitions if the system already has a source of truth.

For each shift, calculate:

A) إجمالي البيع النقدي
= sum of valid cash sales collected in that shift
Use the actual existing invoice/collection model already implemented
Exclude cancelled documents
Avoid duplicate counting

B) إجمالي البيع الآجل / التعاقد
= sum of valid non-cash sales in that shift
This includes deferred / contract / receivable-based sales according to the real payment model in the current app

C) عدد فواتير البيع
= count of distinct valid sales invoices linked to the shift

D) إجمالي المرتجع
= total value of valid returns related to the shift according to the actual return workflow in the app
Inspect the current returns implementation and choose the correct business rule
Then document it clearly before implementation

E) عدد فواتير المرتجع
= count of distinct valid return documents for the shift

F) الصافي
= إجمالي البيع النقدي + إجمالي البيع الآجل/التعاقد - إجمالي المرتجع

G) المبلغ المحول / المسلم للخزنة الرئيسية
= the actual delivered / transferred amount from the shift to the main treasury
Use the existing treasury shift close / delivery model already present in the codebase
Do not invent a fake field if there is already a real source

H) الحالة
= real shift status from the existing source of truth
(open / closed)

QUERY / DATA STRATEGY
==================================================

Use correctness first, then performance.

You must inspect the current schema and identify the real tables/relations for:
- cashier shifts
- sales invoices
- returns
- collections
- treasury shift close / delivered cash / main treasury handover

Build this in storage/repository layer only.
Do not scatter SQL in routes.

Use either:
A) one well-structured aggregate query
OR
B) a small set of safe aggregate queries merged in storage layer

Choose the safer/correcter approach after inspection.

Mandatory query rules:
- aggregate per shift
- avoid duplicate multiplication from joins
- use COUNT(DISTINCT ...) where needed
- use COALESCE for null-safe numeric totals
- stable numeric output
- safe for large datasets

If using multiple aggregate queries:
- one for base shifts
- one for sales aggregates
- one for returns aggregates
- one for treasury transfer aggregates
then merge by shiftId safely in code

Do NOT create a fragile mega-join if it risks double counting.
