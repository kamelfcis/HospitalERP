OBSERVED ISSUES

1) Shift opened by one cashier, but during closing the system suggests transferring invoices to the same cashier / same shift.
This is incorrect.

2) The system has confusion around:
- who opened the shift
- which shifts are truly active
- which user is currently responsible
- which shift should receive pending invoices

3) It is possible for a user to leave a shift open for 1–2 days and return to continue using it normally.
This is incorrect because cashier shifts are day-based and must not remain operational across days without strict rollover rules.

4) There is existing business behavior:
- If more than one shift is open, a shift may close even if it has pending invoices, as long as another valid open shift exists to receive them.
- If only one shift is open and it has pending invoices, closing must be blocked.

This rule must be preserved, but validated more strictly.

CRITICAL BUSINESS RULES

- Shift A → Cashier 1
- Shift B → Cashier 2

- closing report

- audit trail

INVOICE ASSIGNMENT RULE

When two shifts are open at the same time, invoice receipt / cashier assignment must work safely and deterministically.

Required principle:
- invoices may be received normally on both open shifts
- the shift that actually receives/processes the invoice first is the shift that owns it
- ownership must be determined by strong server-side logic, not by weak UI timing only

The plan must propose a STRONG ANTI-CONFUSION assignment model that prevents:
- duplicate claiming
- race conditions
- invoice appearing owned by two shifts
- reassignment without audit trail
- stale session confusion
- refresh-based inconsistency

This must include:
- locking or equivalent concurrency protection
- idempotent claim/receive behavior
- explicit ownership timestamps
- clear source of truth in backend

SHIFT CLOSING RULES

Shift closing must verify at minimum:
- current_user
- current_shift
- target_shift
- active_shifts

Additional strict rules required:

1) target_shift must never equal current_shift
2) target_shift must be actually open and valid
3) if there are pending invoices:
   - closing is allowed only if another valid open shift exists
   - otherwise closing is blocked
4) if only one shift is open and pending invoices exist:
   - do NOT allow close
5) if multiple shifts are open:
   - transfer must go only to a valid receiving shift
   - never to same cashier by mistake
   - never to a stale/inactive shift

DAY-BOUNDARY / STALE SHIFT RULES

Cashier shift is a day-based operational entity.

A shift must not continue normally across multiple days without explicit control.

The review must inspect current logic for:
- shift open date
- business date vs calendar date
- user session vs shift state
- stale open shifts left overnight
- resuming old shifts next day as if nothing happened

The fix plan must define strict behavior for stale shifts, including:
- how to detect stale open shifts
- whether they should be auto-blocked, auto-expired, or require supervisor resolution
- whether login should refuse continuing an old shift
- whether shift close/open rollover is required at day boundary
- how pending invoices behave if a stale shift exists

The system must no longer allow a cashier to casually continue a two-day-old shift.
