REPLIT PROMPT – STORE TO STORE TRANSFER (FEFO-AWARE, NO PRICING CONFLICT)
الميزة: تحويل أصناف بين الأقسام (مخزن → مخزن) مع دعم تاريخ الصلاحية FEFO
Arabic RTL + Peachtree/Sage Web Theme + SQL Server + تصميم محاسبي نظيف بدون كلكعة

====================================================================================================
0) الهدف (Objective)
====================================================================================================
إضافة موديول “تحويل مخزني بين الأقسام” يحقق الآتي:
1) التحويل = حركة مخزنية فقط (OUT من مخزن + IN لمخزن آخر).
2) احترام تاريخ الصلاحية (FEFO) للأصناف التي لها Expiry.
3) نقل التكلفة (Cost) كما هي بدون أي علاقة بأسعار البيع أو أسعار الأقسام.
4) جاهزية كاملة لتقييم المخزون لاحقًا بدون تعديل في التصميم.

⚠️ مهم جدًا:
- التحويل ليس بيع.
- لا يستخدم أسعار الأقسام.
- لا يكوّن إيراد أو مصروف.
- يؤثر فقط على المخزون والتكلفة.

====================================================================================================
1) قواعد العمل (Business Rules) – صارمة
====================================================================================================
A) طبيعة التحويل
- التحويل يتم بين:
  مخزن مصدر (Source Warehouse)
  →
  مخزن وجهة (Destination Warehouse)

- نفس الصنف.
- نفس اللوت (Expiry / Batch إن وجد).
- نفس التكلفة (PurchasePrice).

B) الصلاحية (FEFO)
- إذا الصنف HasExpiry = 1:
  - الخصم يتم من مخزن المصدر حسب:
    ExpiryDate ASC (الأقرب صلاحية أولًا)
  - يمنع التحويل من لوت منتهي الصلاحية.
  - إذا الكمية المطلوبة > رصيد لوت واحد:
    - يتم Split على أكثر من Lot تلقائيًا.

- إذا الصنف HasExpiry = 0:
  - التحويل يتم من الرصيد الإجمالي بدون Expiry.

C) الأسعار
- أسعار الأقسام (ItemDepartmentPrices) ❌ لا تُستخدم.
- سعر البيع ❌ غير معني.
- التكلفة فقط (PurchasePrice على مستوى Lot) ✔️.

D) الأثر المحاسبي
- لا قيد إيراد.
- لا قيد بيع.
- لا تأثير على الربحية.
- فقط إعادة توزيع مخزون بين مخازن.

====================================================================================================
3) قاعدة البيانات (SQL Server) – التعديل المطلوب
====================================================================================================

A) Warehouses (مخازن / أقسام)
Warehouses:
- WarehouseID (PK)
- WarehouseCode
- WarehouseNameAR
- DepartmentID (اختياري)
- IsActive

B) InventoryLots (تعديل إلزامي)
إضافة ربط اللوت بالمخزن:

InventoryLots:
- LotID (PK)
- ItemID (FK)
- WarehouseID (FK -> Warehouses)   <-- جديد (CRITICAL)
- ExpiryDate (DATE NULL)
- ReceivedDate (DATE NOT NULL)
- PurchasePrice (DECIMAL(18,4))
- QtyInMinor (DECIMAL(18,4))
- IsActive
- CreatedAt

Constraints:
- QtyInMinor >= 0
- إذا HasExpiry=0 => ExpiryDate MUST be NULL

Indexes:
- IX_Lots_Item_Warehouse_Expiry (ItemID, WarehouseID, ExpiryDate)
- IX_Lots_Item_Warehouse (ItemID, WarehouseID)

C) InventoryLotMovements (يُستخدم للتحويل)
InventoryLotMovements:
- MovementID (PK)
- LotID (FK)
- WarehouseID (FK)
- TxDate
- TxType (IN / OUT / TRANSFER)
- QtyChangeInMinor
- UnitCost
- ReferenceType = 'TRANSFER'
- ReferenceID = TransferNo
- CreatedAt

====================================================================================================
4) منطق التحويل (Algorithm)
====================================================================================================
Input:
- itemId
- sourceWarehouseId
- destinationWarehouseId
- requiredQtyInMinor
- transferDate

Algorithm:
1) لو item.HasExpiry = 1:
   - Select Lots WHERE:
     - ItemID = itemId
     - WarehouseID = sourceWarehouseId
     - QtyInMinor > 0
     - ExpiryDate >= transferDate
   - Order By ExpiryDate ASC, ReceivedDate ASC
   - Allocate quantities (Split لو لزم)

2) لكل Allocation:
   - OUT Movement من Lot المصدر
   - IN Movement إلى Lot جديد أو موجود في مخزن الوجهة
     بنفس:
       - ExpiryDate
       - PurchasePrice
       - Qty

3) لو الكمية غير كافية:
   - Reject التحويل برسالة واضحة.
