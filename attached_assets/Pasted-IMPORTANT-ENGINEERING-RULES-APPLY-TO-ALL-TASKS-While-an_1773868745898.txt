IMPORTANT ENGINEERING RULES (APPLY TO ALL TASKS)

While analyzing or implementing any fix, follow these mandatory rules:

1. Refactor Discipline
   Do not only patch issues. When needed, refactor code to improve clarity and safety.

Required:

* reduce duplicated logic
* split large functions
* improve naming clarity
* separate business logic from UI logic
* separate API transport from domain logic
* keep invoice logic isolated
* keep cashier logic isolated
* keep pharmacy logic isolated

2. Code Organization
   Maintain clean project structure:

* group related logic
* avoid scattered validation
* move reusable logic to shared services/helpers
* reduce hidden side effects

3. Indexing & Performance Awareness
   Review database performance and indexing.

Especially for:

* invoices
* cashier queues
* pending invoices
* stock lookups
* pharmacy batches
* shift operations
* treasury validation

Suggest indexes when missing.

4. Human-Readable Explanation
   A weaker human maintainer will continue after this work.

For every fix explain:

* what was wrong
* root cause
* what changed
* why solution is safer
* affected files/modules
* DB/index impact

5. Output Format
   Each task must produce:

* findings
* root causes
* refactor opportunities
* affected modules/files
* DB/index impact
* implementation order
* risks
* test checklist
