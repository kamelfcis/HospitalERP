أريدك أن تراجع وتؤكد — ثم تصلح إن لزم — سلوك صلاحيات القسم والمخزن داخل شاشة فاتورة المريض في النظام الحالي، مع الالتزام الصارم بالآتي:

- ابنِ على الموجود ولا تهدم أي جزء شغال.
- لا تعمل حلول شكلية في الواجهة فقط وتترك أصل المشكلة.
- لا تعتمد على إخفاء بصري فقط؛ المطلوب enforcement حقيقي على مستوى الخادم أيضًا.
- أي سلوك متعلق بالقسم أو المخزن أو الخدمات يجب أن يكون متسقًا بين:
  1) الصلاحيات المخزنة للمستخدم
  2) القيم الافتراضية عند فتح الفاتورة
  3) البحث عن الخدمات داخل الفاتورة
  4) الحفظ النهائي على الخادم
- نفّذ Refactor نظيف، Componentization واضح، وفهرسة مناسبة عند الحاجة.
- لا تكسر أي flow قائم في فاتورة المريض أو صلاحيات المستخدمين الحالية.

## الهدف المطلوب التأكد منه
عند دخول "إداري قسم" على شاشة فاتورة مريض:
- يجب أن يكون القسم والمخزن المختصين به محددين تلقائيًا إذا كان هذا هو سلوك النظام الحالي المعتمد من صلاحياته.
- وعند اختيار خدمة داخل الفاتورة، يجب أن يرى فقط الخدمات التابعة لنفس قسمه.
- مثال:
  - إداري العمليات يرى فقط خدمات العمليات
  - إداري المعمل يرى فقط خدمات المعمل
  - إداري الأشعة يرى فقط خدمات الأشعة
- وهذا يجب أن يكون مبنيًا على صلاحياته وربط المستخدم بالقسم والمخزن، وليس مجرد فلترة شكلية قابلة للتجاوز.

## المطلوب منك أولًا — تحليل الواقع الحالي بدقة
راجع فعليًا الكود والـ schema والـ routes والـ storage والـ hooks والـ components المتعلقة بالآتي:

1) كيف يُحدد القسم والمخزن للمستخدم الحالي؟
- هل من user profile؟
- أم role/group mapping؟
- أم user default department / warehouse؟
- أم permission scope؟
- أم من شاشة/إعدادات أخرى؟
حدد المصدر الحقيقي المعتمد حاليًا.

2) عند فتح فاتورة المريض:
- هل يتم تحميل القسم تلقائيًا للمستخدم؟
- هل يتم تحميل المخزن تلقائيًا؟
- هل يمكن للمستخدم تغييره؟
- وهل كل المستخدمين يمكنهم تغيير القسم والمخزن أم فقط فئات معيّنة؟
حلّل السلوك الحالي كما هو، ولا تفترض.

3) راجع ServiceLookup / UnifiedLinesTab / أي service search component مستخدم داخل فاتورة المريض:
- هل يتم تمرير departmentId فعليًا؟
- هل الـ API / resolver يفلتر بالخلفية حسب departmentId؟
- هل هناك أي مكان يسمح بظهور كل الخدمات رغم وجود قسم محدد؟

4) راجع الخادم:
- هل endpoint جلب الخدمات يعتمد فقط على departmentId القادم من الواجهة؟
- أم يعيد التحقق من صلاحيات المستخدم؟
- هل يمكن لمستخدم أن يمرر departmentId مختلف ويجلب خدمات قسم آخر؟
- هل يمكنه حفظ فاتورة بخدمة لا تنتمي للقسم المفترض له؟

5) راجع الحفظ النهائي:
- عند حفظ أو finalize الفاتورة، هل يوجد تحقق أن:
  - القسم المختار مسموح للمستخدم
  - المخزن المختار مسموح للمستخدم
  - كل خدمة على الفاتورة تنتمي للقسم الصحيح أو المسموح
- إن لم يوجد هذا التحقق، أضفه بالطريقة الأنسب حسب بنية النظام الحالية.

## المطلوب منك ثانيًا — القرار الصحيح بناءً على الموجود
بعد التحليل، طبّق السلوك الأنسب حسب تصميم النظام الحالي، مع مراعاة الآتي:

### السيناريو المتوقع
إذا كان المستخدم مرتبطًا بقسم واحد ومخزن واحد فقط:
- افتح الفاتورة بهذا القسم والمخزن تلقائيًا
- اقفل/قيّد التغيير إذا كان ذلك متسقًا مع صلاحياته
- واجعل البحث عن الخدمات يعرض خدمات هذا القسم فقط

إذا كان المستخدم مسموحًا له بأكثر من قسم أو أكثر من مخزن:
- التزم بما يتيحه النظام الحالي من scope
- اعرض فقط الأقسام/المخازن المسموحة له
- وعند اختيار قسم، يجب أن تتفلتر الخدمات فورًا على هذا القسم فقط
- ولا تسمح إطلاقًا بخدمة من قسم خارج القسم المحدد/المسموح

لا تختر سلوكًا نظريًا من عندك؛ اختر الأنسب حسب مسار النظام الحالي.

## المطلوب منك ثالثًا — Enforcement حقيقي وليس UI فقط
أريد التأكد من 4 طبقات معًا:

### 1) Defaulting
- القسم والمخزن يُحمّلان تلقائيًا من صلاحيات المستخدم/ربطه الحالي

### 2) UI Filtering
- شاشة الفاتورة و ServiceLookup تُظهر فقط الخدمات التابعة للقسم المحدد والمسموح

### 3) API/Server Validation
- حتى لو تم العبث من الواجهة أو request يدوي، لا يمكن جلب/حفظ خدمات من قسم غير مسموح

### 4) Save/Finalize Guard
- الخادم يرفض أي فاتورة بها:
  - قسم غير مسموح
  - مخزن غير مسموح
  - خدمة لا تنتمي إلى القسم الصحيح أو خارج scope المستخدم

## المطلوب منك رابعًا — ما يجب فحصه تحديدًا
افحص واعالج - عند الحاجة - النقاط التالية:

1) هل user default department و user default warehouse مجرد convenience أم enforcement؟
2) هل permission scope موجود أصلًا للأقسام والمخازن؟
3) هل هناك owner/admin bypass؟ وإن وجد، حافظ عليه إن كان هذا هو السلوك المقصود.
4) هل هناك فرق بين:
   - رؤية الشاشة
   - إنشاء الفاتورة
   - تعديل القسم
   - تعديل المخزن
   - إضافة الخدمات
   - finalize
   افصل الصلاحيات بوضوح إن كانت موجودة، ولا تخلط view مع create/update/finalize.

5) هل الخدمات القديمة أو الفواتير القديمة يمكن أن تتأثر؟
- حافظ على backward compatibility
- لا تكسر بيانات قديمة

## المطلوب منك خامسًا — التنفيذ المعماري
نفّذ بشكل نظيف:

### Refactor
- لا تضع كل المنطق في component ضخم
- استخرج resolver / guard / helper مركزي لصلاحيات القسم والمخزن والخدمات
- اجعل نفس المنطق يُستخدم في:
  - default values
  - dropdown filtering
  - service lookup
  - save validation

### Componentization
- لو هناك منطق مكرر لتحديد allowed departments / allowed warehouses / allowed services فافصله في hook أو util واضح
- لا تكرر قواعد الأعمال داخل JSX

### Indexing
راجع الفهارس اللازمة فقط إذا كانت الاستعلامات تستخدمها فعلاً، مثل:
- services.department_id
- invoice headers department_id / warehouse_id
- جداول ربط صلاحيات المستخدم بالقسم/المخزن إن وجدت
أضف فقط ما له استخدام فعلي.

### Audit / Logging
بدون مبالغة، أضف logging مناسب لو تم رفض عملية بسبب:
- department scope violation
- warehouse scope violation
- service department mismatch
حتى يسهل تتبع محاولات الاستخدام الخاطئ أو وجود أخطاء ضبط صلاحيات.

## المطلوب منك سادسًا — اختبارات القبول الإلزامية
نفّذ واذكر نتائج اختبار واضحة لهذه السيناريوهات:

1) مستخدم تابع لقسم العمليات فقط → لا يرى إلا خدمات العمليات
2) مستخدم تابع لقسم المعمل فقط → لا يرى إلا خدمات المعمل
3) مستخدم تابع لقسم الأشعة فقط → لا يرى إلا خدمات الأشعة
4) مستخدم له أكثر من قسم → عند تغيير القسم تتغير الخدمات فورًا حسب القسم المختار
5) مستخدم يحاول تمرير request يدوي بخدمة من قسم آخر → الخادم يرفض
6) مستخدم يحاول حفظ فاتورة بمخزن غير مسموح → الخادم يرفض
7) admin/owner — إن كان له bypass مقصود — يستمر سلوكه بدون كسر
8) الفواتير القديمة لا تتكسر
9) الأداء في البحث عن الخدمات لا يتدهور
10) لا يوجد تضارب بين القسم الافتراضي، القسم المعروض، والخدمات المتاحة

## المطلوب منك سابعًا — الرد النهائي
أريد منك في الرد النهائي:

1) Current state الحقيقي بعد فحص الكود
2) هل السلوك المطلوب متحقق حاليًا أم لا
3) ما الفجوات إن وُجدت
4) القرار الذي اخترته ولماذا هو الأنسب للنظام الحالي
5) التعديلات التي نفذتها
6) أين تم enforcement: UI + API + Save/Finalize
7) نتائج اختبارات القبول
8) أي نقاط مؤجلة إن وجدت
9) Done looks like

## تنبيهات صارمة
- لا تكتفِ بتمرير departmentId للبحث وتعتبر المهمة انتهت.
- لا تعتمد على الواجهة فقط.
- لا تسمح لمستخدم يرى خدمات قسم آخر بسبب loophole في API.
- لا تكسر owner/admin bypass إن كان مقصودًا.
- لا تخلط بين default value وبين permission enforcement.
- لا تنفذ حلًا سريعًا ينجح ظاهريًا ويفشل عند الحفظ أو عبر API call مباشر.
- اشتغل بعقلية ERP enterprise: صلاحيات حقيقية، تدفق صحيح، تدقيق، وأمان.

نفّذ بعد فحص الموجود فعليًا، واختر الحل الأنسب حسب مسار النظام الحالي.
