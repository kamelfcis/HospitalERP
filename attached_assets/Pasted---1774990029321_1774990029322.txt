نريد تعديل شاشة "قواعد التغطية" وخصوصًا جزء "اختبار القواعد" + نموذج إضافة/تعديل القاعدة، بحيث يدعم بشكل صحيح ونهائي نوعين مختلفين من الـ domain:

1) خدمات المستشفى (Radiology / Lab / Clinic / OR / ICU ... إلخ)
2) الصيدلية / الأصناف (Drugs / Supplies / Solutions / Consumables ... إلخ)

الهدف:
منع إدخال نص حر في الحقول المرجعية، وجعل الشاشة مناسبة لكل من الخدمات والصيدلية بدون ترقيع لاحق.

========================
المشكلة الحالية
========================

1) حقل "الفئة" نص حر، وهذا يسبب:
- أخطاء كتابة
- عدم تطابق القيم
- قواعد لا تعمل بسبب اختلاف case أو spelling
- خلط بين service categories و item categories

2) حقل "معرف القسم" و"معرف الخدمة" و"الفئة" غير مناسبين للمستخدم النهائي
- تظهر UUIDs أو تتطلب معرفة داخلية
- UX ضعيف جدًا

3) الشاشة حاليًا خليط بين منطق الخدمات ومنطق الصيدلية، وتحتاج فصل domain واضح داخل نفس الواجهة.

========================
المطلوب النهائي
========================

أولًا — إضافة Selector رئيسي يحدد نطاق القاعدة / الاختبار:
- domainType = service | pharmacy

ويظهر للمستخدم باسم عربي واضح:
- خدمات المستشفى
- الصيدلية / الأصناف

هذا الـ selector يجب أن يتحكم في الحقول التي تظهر بعده.

========================
A) عند اختيار "خدمات المستشفى"
========================

يجب أن تدعم الشاشة القواعد والاختبار على مستوى:

1) قسم
2) خدمة
3) فئة خدمة

الحقول يجب أن تكون كلها Dropdown / Combobox، وليست Text Inputs.

### الحقول المطلوبة للخدمات:
- القسم / الوحدة
- الخدمة
- فئة الخدمة

### أمثلة فئات الخدمة:
- RADIOLOGY
- LAB
- CLINIC
- OR
- ICU
- INPATIENT
- OUTPATIENT
- NURSING
(أو ما يعادلها من القيم الفعلية الموجودة في النظام)

### السلوك المطلوب:
1) إذا اختار المستخدم نوع قاعدة متعلق بالقسم:
   - يظهر Dropdown "القسم / الوحدة"
   - يحمل أسماء الأقسام
   - ويُرسل الـ UUID داخليًا فقط

2) إذا اختار المستخدم نوع قاعدة متعلق بالخدمة:
   - يظهر Combobox "الخدمة"
   - يبحث بالاسم
   - ويُرسل serviceId الحقيقي داخليًا

3) إذا اختار المستخدم نوع قاعدة متعلق بفئة الخدمة:
   - يظهر Dropdown "فئة الخدمة"
   - بقيم ثابتة/مرجعية من النظام
   - بدون أي نص حر

========================
B) عند اختيار "الصيدلية / الأصناف"
========================

يجب أن تدعم الشاشة القواعد والاختبار على مستوى:

1) فئة صنف
2) صنف محدد
3) الصيدلية / الوحدة الصيدلية إن لزم

### الحقول المطلوبة للصيدلية:
- الصيدلية / الوحدة
- فئة الصنف
- الصنف

### أمثلة فئات الأصناف:
- DRUG
- SUPPLY
- SOLUTION
- CONSUMABLE
- DISPOSABLE
(أو القيم الحقيقية الموجودة في النظام)

### السلوك المطلوب:
1) إذا اختار المستخدم نوع قاعدة متعلق بفئة صنف:
   - يظهر Dropdown "فئة الصنف"
   - بقيم مرجعية جاهزة
   - بدون نص حر

2) إذا اختار المستخدم نوع قاعدة متعلق بصنف محدد:
   - يظهر Combobox "الصنف"
   - يبحث بالاسم / الكود / الباركود
   - ويرسل itemId داخليًا فقط

3) إذا اختار المستخدم نطاق الصيدلية:
   - يظهر Dropdown "الصيدلية / الوحدة"
   - مثل:
     - صيدلية داخلية
     - صيدلية خارجية
     - مخزن رئيسي أدوية
   - مع تمرير الـ ID الحقيقي داخليًا

========================
C) تعديل أنواع القواعد نفسها
========================

نريد دعم الأنواع التالية بوضوح:

### للخدمات:
- include_department
- exclude_department
- include_service
- exclude_service
- include_service_category
- exclude_service_category
- discount_pct
- fixed_price
- approval_required

### للصيدلية:
- include_item
- exclude_item
- include_item_category
- exclude_item_category
- discount_pct
- fixed_price
- approval_required

مع إظهار أسماء عربية واضحة للمستخدم، مثل:
- تشمل قسم
- تستثني قسم
- تشمل خدمة
- تستثني خدمة
- تشمل فئة خدمة
- تستثني فئة خدمة
- تشمل صنف
- تستثني صنف
- تشمل فئة أصناف
- تستثني فئة أصناف
- خصم بنسبة
- سعر ثابت
- تحتاج موافقة

========================
D) سلوك ديناميكي للحقول
========================

الشاشة يجب أن تُظهر فقط الحقول المناسبة حسب:
1) domainType
2) ruleType

### أمثلة:
- إذا domainType = service و ruleType = include_department
  => يظهر فقط حقل القسم

- إذا domainType = service و ruleType = include_service
  => يظهر فقط حقل الخدمة

- إذا domainType = pharmacy و ruleType = include_item_category
  => يظهر فقط حقل فئة الصنف

- إذا domainType = pharmacy و ruleType = include_item
  => يظهر فقط حقل الصنف

- إذا ruleType = discount_pct
  => يظهر حقل نسبة الخصم
- إذا ruleType = fixed_price
  => يظهر حقل السعر الثابت
- إذا ruleType = approval_required
  => يظهر toggle أو checkbox للموافقة

========================
E) تعديل جزء "اختبار القواعد"
========================

جزء "اختبار القواعد" يجب أن يتبع نفس المنطق الديناميكي، بدل الحقول الحالية الثابتة.

### بدل الحقول الحالية:
- معرف الخدمة
- معرف القسم
- الفئة
- السعر المعلن

يجب أن يصبح الاختبار كالتالي:

#### أولاً: اختيار النطاق
- خدمات المستشفى
- الصيدلية / الأصناف

#### إذا تم اختيار "خدمات المستشفى":
يظهر:
- القسم (Dropdown)
- الخدمة (Combobox)
- فئة الخدمة (Dropdown)
- السعر المعلن

#### إذا تم اختيار "الصيدلية / الأصناف":
يظهر:
- الصيدلية / الوحدة (Dropdown)
- فئة الصنف (Dropdown)
- الصنف (Combobox)
- السعر المعلن

### مهم:
- لا يظهر أي UUID للمستخدم
- لا يوجد أي text input للقيم المرجعية
- كل القيم المرجعية تكون dropdown / combobox

========================
F) مصدر البيانات
========================

يجب تحميل القيم من الجداول/المصادر الحقيقية في النظام:

### للخدمات:
- الأقسام / الوحدات
- الخدمات
- فئات الخدمات

### للصيدلية:
- الصيدليات / الوحدات الصيدلية / المخازن الصيدلية المناسبة
- الأصناف
- فئات الأصناف

إذا كانت بعض الفئات ليست موجودة في جدول مرجعي حاليًا:
- أنشئ مصدرًا مرجعيًا ثابتًا منظمًا في الكود
- ولا تستخدم نصًا حرًا

========================
G) Backward Compatibility
========================

مهم جدًا:
1) القواعد القديمة الموجودة حاليًا تظل تعمل بدون كسر
2) البيانات القديمة التي تحتوي UUID محفوظ تظل قابلة للقراءة والعرض
3) عند فتح قاعدة قديمة:
   - يتم resolve للـ UUID وعرض الاسم البشري المناسب في dropdown
4) لا يتم كسر أي endpoint أو route حالي إن أمكن
5) أي migration يجب أن تكون آمنة

========================
H) UX Improvements
========================

نريد UX واضح جدًا:

1) استبدال كل الحقول النصية المرجعية بـ dropdown/combobox
2) عرض labels عربية واضحة
3) placeholders واضحة
4) لو لم توجد بيانات:
   - "لا توجد خدمات"
   - "لا توجد فئات أصناف"
   - "لا توجد صيدليات"
5) لو domainType = service:
   - لا تظهر حقول الصيدلية
6) لو domainType = pharmacy:
   - لا تظهر حقول الخدمات
7) إضافة helper text بسيط يشرح للمستخدم معنى كل نوع قاعدة

========================
I) Validation
========================

أضف validation صارم:

1) ممنوع حفظ قاعدة بدون القيمة المطلوبة حسب نوعها
2) ممنوع حفظ include_item_category بدون اختيار فئة صنف
3) ممنوع حفظ include_service بدون اختيار خدمة
4) ممنوع حفظ include_department بدون اختيار قسم
5) ممنوع إرسال قيم نصية حرة للفئات
6) السيرفر يجب أن يعيد التحقق، وليس الواجهة فقط

========================
J) Refactor مطلوب
========================

نفّذ التعديل بشكل منظم:
- لا تكدّس المنطق في component واحد
- استخرج helpers/components/hooks عند الحاجة

مثال:
- CoverageDomainSelector
- ServiceCoverageFields
- PharmacyCoverageFields
- CoverageRuleTypeSelector
- CoverageRuleTestPanel

========================
K) النتيجة النهائية المطلوبة
========================

بعد التنفيذ أريد أن تصبح الشاشة كالتالي:

1) أختار:
   - خدمات المستشفى
   أو
   - الصيدلية / الأصناف

2) أختار نوع القاعدة
3) تظهر لي فقط الحقول المناسبة
4) أختار من Dropdowns واضحة
5) أعمل اختبار بدون UUIDs ولا نصوص حرة
6) أحفظ القاعدة بثقة أن النظام فهمها صح

========================
L) تقرير التنفيذ المطلوب
========================

بعد التنفيذ اطبع:
1) الملفات المعدلة
2) أي endpoints أضيفت أو عُدلت
3) مصادر البيانات المستخدمة لكل dropdown
4) كيف تم الحفاظ على backward compatibility
5) ما الذي تغير في Rule Form
6) ما الذي تغير في Test Panel
