أريد عمل Production Hardening لتبويب "الاقتراح الذكي" في التحويلات المخزنية، يشمل توضيح المنطق، تحسين UX، توحيد الحسابات، ومنع أي لبس أو ثغرات.

# الهدف
تثبيت منطق واضح وقابل للفهم للمستخدم:
- التفريق بين "الاحتياج" و"الكمية المقترحة فعليًا"
- توضيح سبب ظهور كل صنف
- توضيح فلتر "استبعاد المكتفية" بأنه مبني على الرصيد مقابل مبيعات الفترة فقط
- ضمان أن التنفيذ لا يتجاوز رصيد المصدر
- الحفاظ على الأداء العالي وعدم كسر أي جزء قائم

---

# 1) توضيح المصطلحات (Rename + UX)
استبدل أي تسمية قد تكون مضللة:

- لا تستخدم "المكتفية" فقط
- استخدم تسمية أوضح:
  "استبعاد الأصناف التي يغطي رصيدها مبيعات الفترة"

وأضف Tooltip واضح:
"يتم الاستبعاد إذا كان رصيد الوجهة الحالي يغطي مبيعات الفترة المحددة، دون اعتبار حد أدنى أو مخزون أمان."

---

# 2) فصل المفاهيم الحسابية (مهم جداً)
عرّف وأظهر بوضوح:

- الاحتياج (Need):
  need = max(0, destination_sales - destination_stock)

- الكمية المقترحة فعلياً (Suggested Transfer):
  suggested_qty = min(need, source_stock)

المطلوب:
- إظهار العمودين بشكل منفصل إن أمكن:
  - "الاحتياج"
  - "المقترح للتحويل"
- أو على الأقل توضيح الفرق في UI

---

# 3) توضيح قص الكمية حسب رصيد المصدر
إذا كان:
- source_stock < need

أضف Badge أو Indicator:
- "المصدر غير كافٍ"
- أو "المقترح مقصوص حسب رصيد المصدر"

ولا تترك المستخدم يتساءل لماذا الكمية أقل من العجز.

---

# 4) تعريف سبب ظهور الصنف (Suggestion Reason)
لا تستخدم "جديد" بشكل عام.

أضف field أو badge باسم:
suggestionReason

قيمه تكون:
- sales_gap → عجز مبيعات
- destination_zero_available_in_source → الوجهة صفر والمصدر به رصيد

UI:
- "عجز مبيعات"
- "الوجهة صفر"
- أو "عاد للتوفر" (اختر تسمية واضحة)

---

# 5) تعريف "الوجهة صفر / عاد للتوفر"
الصنف يظهر بهذه الحالة إذا:
- source_stock > 0
- destination_stock = 0
- ولا يشترط وجود مبيعات حديثة

مهم:
- هذا سبب للظهور فقط
- لا يعني بالضرورة اقتراح كمية كبيرة تلقائيًا
- راجع سلوك الكمية في هذه الحالة (0 أو قيمة محدودة أو leave to user)

---

# 6) زر "ملء الكميات المقترحة"
وضّح سلوكه:

- هل يملأ الصفحة الحالية فقط؟
- أم كل النتائج؟

المطلوب:
- إذا كان يملأ الصفحة فقط:
  أضف نص واضح:
  "يتم ملء الكميات للأصناف الظاهرة فقط"

- تأكد أن الكمية المملوءة:
  = suggested_qty (بعد قصها برصيد المصدر)

---

# 7) دقة تعريف "مبيعات الوجهة"
راجع الاستعلام:

- تأكد أن:
  tx_type = out
  AND reference_type IN (sales_invoice, patient_invoice)

- استبعد:
  - مرتجعات
  - تحويلات
  - تسويات
  - أي حركة ليست طلب فعلي

---

# 8) الأداء والفهارس
- راجع أن الـ CTE لا يسبب full scan غير ضروري
- تأكد من وجود index مناسب على:
  inventory_lot_movements (tx_type, reference_type, tx_date, item_id, warehouse_id)
  inventory_lots (warehouse_id, item_id)

- لا تضف فهارس عشوائية

---

# 9) Refactor
- استخرج دوال حساب:
  - need
  - suggested_qty
  - suggestionReason

في helper مشترك (frontend/backend حسب الحاجة)

- لا تكرر نفس المعادلات في أكثر من مكان

---

# 10) اختبارات إلزامية
نفّذ واثبت:

1) source_stock > need
→ suggested = need

2) source_stock < need
→ suggested = source_stock + يظهر badge

3) destination_stock >= sales
→ الصنف يُستبعد عند تفعيل الفلتر

4) destination_stock = 0 + source_stock > 0
→ يظهر كـ "الوجهة صفر"

5) pagination:
→ fill يطبق فقط على الصفحة الحالية (أو حسب التصميم)

6) أصناف بدون مبيعات:
→ تظهر فقط حسب شرط "الوجهة صفر"

---

# 11) المطلوب في الرد
أريد تقرير واضح:

1) MODIFIED FILES
2) RENAMING & UX CHANGES
3) CALCULATION LOGIC (need / suggested)
4) SUGGESTION REASONS IMPLEMENTED
5) FILTER BEHAVIOR
6) PERFORMANCE / INDEX REVIEW
7) EDGE CASES
8) TEST RESULTS

---

# قيود
- لا تغيّر المنطق الأساسي الحالي
- لا تكسر التحويلات أو أي شاشة أخرى
- لا تخلط بين "سبب الظهور" و"الكمية المقترحة"
- لا تعتمد على UI فقط — الحسابات يجب أن تكون موحدة

نفّذ كـ production-grade refactor + UX clarification، وليس تعديل شكلي.
