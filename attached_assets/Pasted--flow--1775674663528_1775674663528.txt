أريد تنفيذ تطوير معماري/وظيفي قوي جدًا فوق النظام الحالي، بدون هدم الموجود، وبدون كسر أي flow قائم، وبأسلوب enterprise-grade clean architecture. المطلوب ليس patch سريع، بل تنفيذ نظيف جدًا، منظّم، عالي الأداء، قابل للتوسع، ومبني على refactor وcomponentization وفهرسة وتحسين استعلامات بشكل قوي.

==================================================
الهدف الرئيسي
==================================================

أريد أن تصبح شاشة "ملف المريض" هي المظلة الرئيسية الكاملة للمريض داخل النظام، وتحتوي داخليًا على الجزء المالي الخاص بالفواتير المجمعة، مع مرونة قوية جدًا في العرض والطباعة والتقارير.

داخل "ملف المريض" أريد أن أستطيع:

1) رؤية البيانات الأساسية للمريض
2) رؤية الزيارات Visits
3) رؤية البيانات الطبية الأساسية/التشخيصات إن كانت موجودة في النظام الحالي
4) رؤية الفواتير الأصلية لكل قسم
5) رؤية "فاتورة مجمعة" داخل ملف المريض
6) طباعة الفاتورة المجمعة بأكثر من مستوى وبأكثر من شكل
7) كل هذا يتم فوق الموجود، بدون هدم، وبكود نظيف جدًا، سريع جدًا، مفهرس، ومقسّم إلى components/services/helpers واضحة

==================================================
أولًا: مبادئ التنفيذ الإلزامية
==================================================

نفّذ وفق المبادئ التالية إلزاميًا:

- لا تهدم أي flows حالية تعمل
- ابْنِ فوق الموجود
- أي refactor يكون آمنًا تدريجيًا
- routes تكون thin جدًا
- انقل business logic إلى services/helpers/storage layers
- لا تضع logic ثقيل داخل React pages
- افصل UI container عن presentational components
- افصل fetching logic عن rendering logic
- افصل print/export builders عن UI
- افصل aggregation logic عن screen components
- افصل permission logic عن rendering قدر الإمكان
- لا تسمح بأي monolithic component كبير
- لا تسمح بأي file ضخم مسؤول عن كل شيء
- أي query ثقيلة راجعها، وقلّل N+1، واعمل batching، واعمل indexing مناسب
- أي rendering كبير اعمل له memoization وتثبيت للـ props والcallbacks
- أي derived data مكلفة اعمل لها memoized selectors/helpers
- حافظ على backward compatibility

==================================================
ثانيًا: المطلوب الوظيفي بالتحديد
==================================================

أريد تطوير شاشة "ملف المريض" لتصبح Patient Master Workspace متكاملة، وبداخلها تبويب/قسم مالي قوي للفواتير المجمعة.

--------------------------------------------------
A) Patient File as Main Workspace
--------------------------------------------------

اجعل "ملف المريض" هو الشاشة الرئيسية الجامعة للمريض، مع الحفاظ على الموجود، وتنظيمه تحت tabs/sections واضحة جدًا على الأقل كالتالي:

1) Overview / نظرة عامة
- بيانات المريض الأساسية
- كود المريض
- ملخص سريع للحالة الحالية
- نوع الحالة الحالية إن أمكن: داخلي / خارجي
- ملخص مالي سريع:
  - إجمالي الفواتير
  - المدفوع
  - المتبقي
- آخر زيارة / آخر Admission إن وجدت

2) Visits / الزيارات
- عرض كل الزيارات المرتبطة بالمريض
- داخلي / خارجي
- التواريخ
- الحالة
- admission إن وجد
- visit grouping إن وجد
- لا تغيّر المنطق الحالي جذريًا إن لم يكن هناك Visit entity كاملة، لكن نظّم العرض فوق الموجود الحالي

3) Medical / طبي
- اعرض التشخيصات أو البيانات الطبية الأساسية إن كانت موجودة بالفعل في النظام
- لا تخترع clinical model جديد الآن
- فقط استهلك الموجود إن وجد، ونظّم عرضه

4) Original Invoices / الفواتير الأصلية
- عرض كل فواتير الأقسام الأصلية المرتبطة بالمريض أو بالزيارة
- إقامة
- معمل
- أشعة
- خدمات
- صيدلية
- عمليات
- مع الحالة والمبالغ والمدفوع والمتبقي

5) Consolidated Invoice / الفاتورة المجمعة
- هذا أهم جزء
- يجب أن يعرض التجميع المالي بشكل احترافي جدًا داخل ملف المريض

6) Payments / المدفوعات
- كل المدفوعات المتعلقة بالمريض/الفواتير حسب الموجود حاليًا
- الخزنة
- الطريقة
- المستخدم
- التاريخ
- الفاتورة المرتبطة إن كانت موجودة

7) Statement / كشف الحساب
- عرض مالي واضح ومفهوم
- إجمالي مستحق
- إجمالي مدفوع
- رصيد متبقٍ
- على مستوى المريض
- ومع دعم breakdown حسب الزيارة إن أمكن من الموجود الحالي

--------------------------------------------------
B) Consolidated Invoice inside Patient File
--------------------------------------------------

أريد داخل ملف المريض تبويب/قسم متقدم جدًا للفواتير المجمعة بحيث يدعم 4 مستويات عرض وطباعة على الأقل:

### View/Print Mode 1 — Summary by Visit
مثال:
- Visit 1
  - إجمالي الإقامة
  - إجمالي المعمل
  - إجمالي الأشعة
  - إجمالي الخدمات
  - إجمالي الأدوية
  - إجمالي المستهلكات
  - إجمالي الزيارة
  - المدفوع
  - المتبقي

- Visit 2
  - نفس الفكرة

هذا المستوى مهم جدًا.

### View/Print Mode 2 — Summary by Department
مثال:
- الإقامة
- المعمل
- الأشعة
- العمليات
- الصيدلية
- الخدمات

كل قسم يظهر:
- عدد البنود/الفواتير
- الإجمالي
- المدفوع
- المتبقي

### View/Print Mode 3 — Summary by Classification
مثال:
- خدمات
- أدوية
- مستهلكات
- أجهزة/معدات إن وجدت
- إقامة

كل classification يظهر:
- عدد البنود
- الإجمالي
- الخصم
- الصافي
- المدفوع
- المتبقي

### View/Print Mode 4 — Detailed Line-by-Line
تفصيلي كامل جدًا:
- التاريخ
- رقم الزيارة/المصدر إن وجد
- القسم
- نوع البند
- اسم البند
- الكمية
- السعر
- الخصم
- الصافي
- المدفوع/المتبقي إن كان مرتبطًا
- الفاتورة الأصلية
- source_type / source_id لو موجودين داخليًا

هذا هو التقرير التفصيلي الكامل.

--------------------------------------------------
C) Printing & Reporting Flexibility
--------------------------------------------------

أريد نظام طباعة وتقارير مرن جدًا داخل ملف المريض، وليس مجرد print للشاشة.

نفّذ بنية طباعة وتقارير تسمح بـ:

1) طباعة الفاتورة المجمعة على مستوى الزيارة
2) طباعة الفاتورة المجمعة على مستوى الأقسام
3) طباعة الفاتورة المجمعة على مستوى التصنيفات:
   - خدمات
   - أدوية
   - مستهلكات
   - إقامة
4) طباعة تفصيلية بكل البنود
5) اختيار ما إذا كانت الطباعة:
   - تشمل المدفوع والمتبقي
   - تشمل فقط الإجماليات
   - تشمل الفواتير الأصلية المرجعية
   - تشمل بيانات المريض الكاملة أو مختصرة

مهم جدًا:
- لا تجعل منطق التقرير داخل الصفحة مباشرة
- أنشئ report builders / print formatters / aggregation services نظيفة جدًا
- اجعل إضافة نمط طباعة جديد لاحقًا سهلة جدًا

--------------------------------------------------
B) Backend / Service Layer Refactor
--------------------------------------------------

أريد منطق التجميع والتقرير والطباعة يخرج من الـ routes والشاشات إلى service layer نظيفة.

مثال منطقي مطلوب:

- patient-visit-aggregation-service
- patient-consolidation-service
- patient-financial-summary-service
- patient-print-report-service

أي route حالي يشارك في هذا الملف:
- خفّفه
- اجعله orchestration فقط
- business logic تذهب إلى services
- queries المعقدة تذهب إلى storage/repository layer

--------------------------------------------------
C) Data Aggregation Rules
--------------------------------------------------

أريدك أن تعتمد على الموجود في النظام الحالي، وتبني فوقه بدون اختراع يكسّر المنطق القائم.

قواعد التجميع المطلوبة:

1) لا تحذف الفواتير الأصلية
2) التجميع يكون عرضيًا/تحليليًا وتقاريريًا فوق الأصل
3) إذا كان هناك visit grouping أو admission grouping حالي، استهلكه كما هو
4) إن لم توجد visit entity كاملة، ابْنِ fallback واضحًا من الموجود الحالي
5) استخدم source_type / source_id إن كانت متاحة لربط السطور بمصدرها
6) أي classification موجود بالفعل استعمله، ولا تخلق classification جديدة بلا داع
7) أي medical diagnosis أو notes أو encounter info موجودة، فقط اعرضها بشكل منظم

==================================================
رابعًا: الأداء Performance إلزامي
==================================================

هذا الجزء مهم جدًا. أريد تحسين أداء حقيقي وليس شكلي.

--------------------------------------------------
A) Query Performance
--------------------------------------------------

راجع كل الاستعلامات الخاصة بـ:
- patient file
- admissions
- visits
- invoices
- lines
- payments
- consolidated views
- statement summaries

وحقق الآتي:
- منع N+1
- batch fetch wherever possible
- pre-aggregate when safe
- use selective joins only
- لا تجلب أعمدة لا نحتاجها
- pagination أو windowing إذا كانت هناك جداول تفاصيل كبيرة
- lazy loading للتبويبات الثقيلة إن لزم
- server-side filtering/sorting للتفاصيل الكبيرة عند الحاجة

--------------------------------------------------
B) Indexing
--------------------------------------------------

أضف/راجع الفهارس اللازمة للاستعلامات الفعلية التي ستُستخدم في Patient File وConsolidated Views.

اهتم خصوصًا بالفهارس على:
- patient_id
- admission_id
- visit_group_id
- invoice header foreign keys
- line source references
- status fields المستخدمة في filters
- created_at / service_date / invoice_date لو تستعمل في الترتيب
- payment header references
- أي composite indexes لازمة لشاشات العرض الجديدة

لكن:
- لا تضف indexes عشوائية
- اجعل كل index مبررة بالاستعلامات الفعلية
- اشرح بإيجاز لماذا كل index مضافة
- حافظ على التوازن بين القراءة والكتابة

--------------------------------------------------
C) Frontend Rendering Performance
--------------------------------------------------

أريد:
- memoization محترمة
- stable callbacks
- avoid unnecessary rerenders
- split heavy tables
- virtualized rendering إذا لزم
- tab-level data fetching when appropriate
- no expensive transformations on every render
- derived data moved to memoized helpers/selectors

سادسًا: القيود المهمة جدًا
