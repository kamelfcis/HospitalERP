أريدك أن تقوم بمراجعة نهائية (Audit Review) على تنفيذ نظام صلاحيات الأقسام والمخازن داخل فاتورة المريض، بعد التعديلات الأخيرة، مع التركيز على الجودة، الأمان، والتوافق مع تصميم ERP احترافي.

❗ مهم:
- لا تنفذ أي كود جديد إلا إذا اكتشفت مشكلة حقيقية.
- الهدف هو التقييم + تحسينات دقيقة فقط (Hardening / Refinement).
- لا تكسر أي flow قائم.
- لا تعيد كتابة أجزاء سليمة.
- اعتمد على الموجود وراجع بعمق.

━━━━━━━━━━━━━━━━━━━━━━━
🎯 الهدف من المراجعة
التأكد أن النظام يحقق فعليًا:

1) المستخدم يرى فقط:
   - الأقسام المسموح بها
   - المخازن المسموح بها
   - الخدمات التابعة للقسم المحدد

2) لا يمكن تجاوز الصلاحيات عبر:
   - API call مباشر
   - تعديل request يدوي
   - تغيير UI state

3) الحفظ والاعتماد (Finalize) محمي بالكامل

━━━━━━━━━━━━━━━━━━━━━━━
🔍 المطلوب منك فحصه بدقة

### 1) توحيد مصدر الحقيقة (Single Source of Truth)
- هل يوجد Resolver مركزي واحد يحدد:
  - allowedDepartmentIds
  - allowedWarehouseIds
- هل نفس المنطق يُستخدم في:
  - auth/me
  - UI filtering
  - API validation
  - save/finalize

❗ لو المنطق متكرر في أكثر من مكان → اقترح توحيده بدون كسر.

━━━━━━━━━━━━━━━━━━━━━━━
### 2) الفرق بين Default vs Enforcement
تأكد أن:
- default department / warehouse ≠ permission
- النظام لا يعتمد على default فقط
- enforcement الحقيقي يتم على الخادم

❗ لو في أي مكان يعتمد على default بدل permission → خطأ لازم يتصلح.

━━━━━━━━━━━━━━━━━━━━━━━
### 3) UI Layer
راجع:
- visibleDepartments
- visibleWarehouses
- ServiceLookup

تأكد:
- لا تظهر أي قيمة خارج scope
- تغيير القسم يفلتر الخدمات فورًا
- لا يوجد cache قديم يعرض خدمات خاطئة

❗ لا تعتمد على UI فقط.

━━━━━━━━━━━━━━━━━━━━━━━
### 4) API Layer (الأهم)
راجع endpoints:

- GET /api/services
- CREATE invoice
- UPDATE invoice
- FINALIZE invoice

تأكد:
- رفض أي department خارج scope → 403
- رفض أي warehouse خارج scope → 403
- رفض أي service لا ينتمي للقسم الصحيح

❗ اختبر سيناريو:
User يرسل request يدوي بقيم مختلفة → يجب الرفض.

━━━━━━━━━━━━━━━━━━━━━━━
### 5) Service-Department Consistency
تأكد:
- كل service لها department واضح
- أو لو departmentId = null → تعامل كخدمة عامة intentional

❗ لو في ambiguity → اقترح حل واضح.

━━━━━━━━━━━━━━━━━━━━━━━
### 6) Admin / Owner Bypass
تأكد:
- bypass يعمل فقط لمن يجب
- لا يوجد escalation لمستخدم عادي

❗ أي loophole هنا = خطر.

━━━━━━━━━━━━━━━━━━━━━━━
### 7) UX Refinement (مهم)
راجع القرار التالي:

إذا:
- allowedDepartmentIds.length === 1
- allowedWarehouseIds.length === 1

هل الأفضل:
- disable الاختيار؟
- أم إخفاء الحقل؟
- أم تركه قابل للتغيير داخل scope؟

❗ اختر الأنسب بناءً على:
- وضوح المستخدم
- منع الخطأ
- عدم التعقيد

━━━━━━━━━━━━━━━━━━━━━━━
### 8) Logging & Audit
راجع:

هل يتم تسجيل:
- department_scope_violation
- warehouse_scope_violation
- service_department_mismatch

❗ لو غير موجود → اقترح logging خفيف بدون تحميل زائد.

━━━━━━━━━━━━━━━━━━━━━━━
### 9) Performance
راجع:

- هل scope-guard يضيف queries إضافية؟
- هل يوجد N+1؟
- هل يمكن caching آمن؟

❗ لا تضحي بالأداء في شاشة الفاتورة.

━━━━━━━━━━━━━━━━━━━━━━━
### 10) Backward Compatibility
تأكد:
- الفواتير القديمة تعمل
- لا يوجد breaking change
- لا يوجد اعتماد على حقول جديدة غير موجودة سابقًا

━━━━━━━━━━━━━━━━━━━━━━━
🧪 اختبارات إلزامية (لازم تتأكد منها)

1) مستخدم قسم واحد → يرى خدمات قسمه فقط
2) مستخدم متعدد الأقسام → تتغير الخدمات حسب القسم
3) request يدوي بقسم آخر → 403
4) محاولة حفظ بمخزن غير مسموح → 403
5) service من قسم مختلف → رفض
6) admin → يعمل بدون قيود
7) الأداء مستقر

━━━━━━━━━━━━━━━━━━━━━━━
📦 المطلوب في الرد النهائي

1) هل التنفيذ الحالي سليم 100% أم لا
2) أي ثغرات أو مخاطر متبقية
3) أي تحسينات بسيطة (Refinement)
4) هل architecture نظيف أم يحتاج توحيد
5) هل يوجد تكرار في المنطق
6) هل يوجد أي risk مستقبلي
7) Done looks like

━━━━━━━━━━━━━━━━━━━━━━━
⚠️ تنبيهات صارمة

- لا تكتب كود إلا عند وجود مشكلة حقيقية
- لا تعيد بناء النظام
- لا تضيف تعقيد غير ضروري
- لا تعتمد على UI
- ركّز على Server Authority
- ركّز على Auditability
