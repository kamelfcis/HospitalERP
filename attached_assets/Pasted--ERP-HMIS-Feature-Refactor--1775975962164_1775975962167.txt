أنت تعمل على نظام ERP/HMIS حيّ وقائم بالفعل، وأي تعديل أو Feature جديدة أو Refactor يجب أن يتم بمنتهى الحذر، مع الحفاظ الكامل على السلوك الحالي وعدم كسر أي Flow موجود.

**قاعدة معمارية إلزامية وثابتة لا يجوز تجاوزها إطلاقًا:**

* الحد الأقصى لأي ملف كود هو **300 سطر فقط**.
* لو التعديل سيجعل الملف يقترب من 300 سطر أو يتجاوزها: **يجب التقسيم فورًا قبل الإضافة**.
* ممنوع تمامًا إنشاء ملفات monolithic أو تجميع أكثر من مسؤولية في ملف واحد.
* كل كود جديد أو تعديل جديد يجب أن يلتزم بـ:

  1. **Refactor-first mindset**
  2. **Single responsibility**
  3. **Componentization / modularization**
  4. **Thin routes / thin controllers**
  5. **Service layer واضحة**
  6. **Storage/repository layer واضحة**
  7. **Indexing and performance awareness**
  8. **Backward compatibility**
  9. **No hidden side effects**
  10. **No duplication**

## الهدف الإجباري

نفّذ أي طلب جديد أو تعديل جديد **بذكاء معماري** يضمن:

* عدم كسر النظام الحالي.
* عدم كسر الكود الجديد.
* عدم تعقيد المشروع.
* عدم تضخم الملفات.
* سهولة الصيانة لاحقًا.
* سهولة الاختبار والمراجعة.
* قابلية التوسع.

## قواعد إلزامية قبل تنفيذ أي تعديل

قبل كتابة أي كود:

1. افهم الـ flow الحالي كاملًا.
2. حدّد الملفات المتأثرة.
3. لو وجدت ملفًا كبيرًا أو مرشحًا للتضخم: **قسّمه أولًا**.
4. لا تضف logic جديد داخل ملف مزدحم.
5. لا تضع business logic داخل routes أو UI components الثقيلة.
6. لا تربط بين أكثر من concern في نفس الملف.
7. لا تعمل quick patch إذا كان سيزيد التعقيد.
8. أي حل لازم يكون **production-safe** وليس مجرد حل مؤقت.

## قاعدة الـ 300 سطر — Mandatory

هذه القاعدة غير قابلة للتفاوض:

* أي ملف جديد: **يفضل أقل من 200–250 سطر**.
* **300 سطر هو السقف النهائي المطلق**.
* إذا وصل الملف إلى 260+ سطر أثناء التنفيذ، اعتبره إنذار مبكر وابدأ التقسيم.
* لو احتاجت الميزة أكثر من ذلك:

  * استخرج helpers
  * استخرج hooks
  * استخرج services
  * استخرج validators
  * استخرج mappers
  * استخرج query builders
  * استخرج sub-components
  * استخرج constants/types/enums إلى ملفات مستقلة
* لا تؤجل التقسيم إلى ما بعد الانتهاء؛ **التقسيم جزء من التنفيذ نفسه**.

## قواعد الـ Refactor الإجباري

أي تنفيذ يجب أن يراعي التالي:

* لو الملف الحالي متضخم: ابدأ بـ refactor آمن قبل إضافة المنطق الجديد.
* refactor لازم يكون محافظًا على نفس السلوك functional behavior.
* لا تغيّر contract أو API أو shape البيانات إلا لو مطلوب صراحة.
* لو احتجت تعديل واسع:

  * قسّمه إلى وحدات صغيرة
  * حافظ على façade أو orchestrator واضح
  * اجعل نقطة الاستدعاء الرئيسية بسيطة جدًا
* لا تكرر logic موجود؛ أعد استخدامه أو انقله لوحدة مشتركة.

## قواعد الـ Componentization

في الفرونت:

* ممنوع component واحد يبتلع:

  * fetching
  * state management
  * rendering
  * calculations
  * modals
  * tables
  * permissions
  * side effects
    كلها معًا.
* افصل:

  * page container
  * toolbar/header
  * filters
  * data table
  * row component
  * forms
  * dialogs
  * hooks
  * utils
* أي شاشة كبيرة يجب تقسيمها إلى subcomponents واضحة.
* أي hook كبيرة تُقسم إلى hooks أصغر حسب المسؤولية.
* أي calculations معقدة تُنقل خارج الـ component.

في الباك:

* route file يجب أن يكون thin جدًا.
* ممنوع حشر business rules داخل route.
* route مسؤوليته:

  * parse request
  * auth/permission check
  * call service
  * return response
* business rules داخل service layer.
* database access داخل storage/repository layer.
* validation المنطقي في validators/helpers منظمة.

## قواعد الفهرسة والأداء

أي feature جديدة أو تعديل على query أو report أو listing أو search لازم يراعي الأداء من البداية:

* فكّر في الفهارس المطلوبة قبل اعتماد الحل.
* أضف indexing لأي:

  * foreign keys كثيرة الاستخدام
  * filtering columns
  * sorting columns
  * join keys
  * unique business keys عند الحاجة
  * partial/composite indexes عند اللزوم
* ممنوع إضافة query ثقيلة دون مراجعة أثرها.
* قلل N+1 queries.
* استخدم batching where appropriate.
* لا تعتمد على تحميل كل البيانات ثم الفلترة في الذاكرة إذا كان يمكن تنفيذها بكفاءة في DB.
* أي report أو search أو list جديدة يجب أن تكون قابلة للتوسع على production data.

## قواعد الأمان وعدم كسر النظام

لا تنفذ أي تعديل بشكل يهدد الاستقرار:

* حافظ على backward compatibility.
* لا تكسر existing endpoints.
* لا تكسر current DB flows.
* لا تغيّر أسماء أو contracts مستخدمة بدون ضرورة شديدة.
* لا تحذف كود قديم إلا لو تأكدت 100% أنه unused وآمن الحذف.
* لو هناك سلوك قديم يجب الحفاظ عليه: ابنِ فوقه ولا تهده.
* استخدم feature-safe refactor.
* أي تغيير حساس لازم يكون atomic قدر الإمكان.
* أي خطوة قد تؤدي لانحراف في السلوك الحالي يجب تجنبها أو عزلها بعناية.

## قواعد الذكاء في التنفيذ

نفّذ بذكاء وليس بشكل حرفي أعمى:

* لا تقسّم التقسيم لمجرد التقسيم؛ قسّم حسب المسؤولية.
* لا تخلق ملفات كثيرة بلا معنى؛ اجعل كل ملف له دور واضح.
* لا تعقّد الاستدعاءات.
* لا تبني abstraction مبالغ فيها.
* لا تضحّي بالوضوح من أجل الشكل.
* الهدف: **بساطة قوية وقابلة للصيانة**.

## أسلوب التنفيذ الإجباري لكل مهمة

مع كل طلب جديد:

1. حلّل الوضع الحالي.
2. حدّد أين يجب الإضافة أو التعديل.
3. لو هناك خطر تضخم ملف: قم بالتقسيم أولًا.
4. نفّذ التعديل عبر وحدات صغيرة واضحة.
5. راعِ الفهارس والأداء.
6. راعِ عدم كسر النظام.
7. راعِ قابلية الصيانة.
8. في النهاية قدّم ملخصًا واضحًا يبيّن:

   * ما الذي تغيّر
   * لماذا هذا التقسيم
   * كيف تم الحفاظ على السلوك الحالي
   * ما الفهارس أو تحسينات الأداء التي أُضيفت
   * تأكيد أن لا ملف تجاوز 300 سطر

## ممنوعات صريحة

ممنوع:

* ملفات فوق 300 سطر
* monolith files
* god components
* god services
* business logic في routes
* heavy logic داخل UI render
* duplication
* quick hacks
* silent behavior changes
* broad unsafe rewrites
* كسر الواجهات الحالية دون ضرورة
* تجاهل indexing في أي feature مؤثرة على القراءة/البحث/التقارير

## معايير القبول الإلزامية

لا تعتبر المهمة منتهية إلا إذا تحققت كل الشروط التالية:

* لا يوجد أي ملف تعدى 300 سطر
* الكود الجديد مقسم بوضوح
* الـ business logic في المكان الصحيح
* الـ UI/components غير متضخمة
* الـ routes رفيعة
* الـ services/storage واضحة
* الأداء مأخوذ في الاعتبار
* الفهارس المناسبة تمت مراعاتها
* لا يوجد كسر للـ flows الحالية
* الكود أوضح وأسهل صيانة من قبل

## صيغة الالتزام النهائي

اعتبر هذه القاعدة **Policy ثابتة ودائمة** على كل ما ستنفذه لاحقًا داخل المشروع:
**"Maximum file size = 300 lines, mandatory refactor, mandatory componentization, mandatory indexing awareness, zero tolerance for monolithic growth or unsafe implementation."**

وطبّقها دائمًا تلقائيًا على أي كود جديد أو تعديل جديد، حتى لو لم يُطلب ذلك صراحة في كل مرة.
