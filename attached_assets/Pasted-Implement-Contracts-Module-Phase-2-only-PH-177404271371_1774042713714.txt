Implement Contracts Module — Phase 2 only.

==================================================
PHASE 2 GOAL
==================================================

Build the deterministic server-side Coverage / Pricing Rules Engine.

This is the core intelligence of the contracts module.

Every time a service is added anywhere in the system, the backend must evaluate:
- the active contract
- the matching rule
- the contract price
- company share
- patient share
- coverage status
- approval requirement

This result must be stamped on the invoice/service line.

Do NOT implement claims settlement in this phase.
Do NOT move accounting settlement here beyond storing the stamped line data.

==================================================
PHASE 2 — RULE EVALUATOR
==================================================

Create:
server/lib/contract-rule-evaluator.ts

Export:
- evaluateContractForService(params)
- batchEvaluate(params[])

EvaluationResult must contain:
- covered
- coverageStatus
- contractPrice
- companyShareAmount
- patientShareAmount
- matchedRuleId
- matchedRuleName
- explanation
- companyCoveragePct

Rules:
- evaluator must be pure
- no DB writes inside evaluator
- preloaded contract/rules preferred for performance

Matching precedence:
- exact service
- exact department
- service category
- global/default

Priority:
- lower priority number = higher priority

Rule behavior:
- include_* => covered, price from base list or listPrice, shares by companyCoveragePct
- exclude_* => not covered
- discount_pct => reduced contract price
- fixed_price => fixed contract price
- approval_required => approval_required status
- global_discount => fallback discount

Fallback if no rule:
- use base price list if available
- else listPrice
- apply companyCoveragePct if defined

Return human-readable explanation for audit/debug/support.

Validate contract and services properly.
