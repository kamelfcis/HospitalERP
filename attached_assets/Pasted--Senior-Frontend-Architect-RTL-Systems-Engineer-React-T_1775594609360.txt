أنت الآن Senior Frontend Architect + RTL Systems Engineer داخل مشروع React/TypeScript كبير، ومطلوب منك **تنفيذ حل RTL system-wide من الملف/الملفات الأساسية** بحيث:

* أي شاشة قديمة تلتزم تلقائيًا بالنمط الصحيح
* أي شاشة جديدة ترث نفس السلوك تلقائيًا
* لا نحتاج ترقيعات محلية متكررة لكل صفحة
* لا ينكسر أي layout قائم صحيح
* لا يتم الاعتماد على حلول عشوائية داخل كل شاشة

# الهدف

تطبيق **RTL architecture حقيقي على مستوى النظام** من الـ root layout + CSS system + shared wrappers بحيث يكون:

* النصوص RTL
* الـ toolbars RTL
* الـ filter rows RTL
* الـ grids/tables RTL
* الـ pagination RTL
* أزرار الأيقونات والنصوص تظهر بترتيب صحيح
* أي صفحة جديدة تستخدم الـ shared classes أو layout الأساسي ترث نفس السلوك مباشرة

---

# المطلوب تنفيذه

## 1) تثبيت الاتجاه العام من الجذر

راجع الملف الأساسي المسؤول عن الـ app shell / layout / root container مثل:

* AppLayout
* main app wrapper
* global layout
* root container
* index.css / globals.css

ونفّذ direction بشكل صريح على المستويات الأساسية التالية:

* `html`
* `body`
* `#root`
* العنصر الأساسي الذي يحتوي صفحات التطبيق مثل `<main>`

الهدف:

* إنهاء الاعتماد على وراثة غير مضمونة
* جعل RTL صريحًا ومضمونًا على مستوى النظام

لكن:

* لا تضع تغييرات قد تكسر مكونات تحتاج LTR جزئيًا
* اجعل الأساس RTL، مع إمكانية استثناء موضعي لاحقًا عند الحاجة

---

## 2) بناء System-wide RTL rules للـ shared UI patterns

بدلاً من إصلاح كل شاشة يدويًا، ابحث عن الـ shared classes المستخدمة عبر الشاشات مثل:

* `.peachtree-toolbar`
* `.peachtree-grid`
* `.peachtree-filters`
* `.peachtree-pagination`
* أي classes مشتركة مشابهة

ونفّذ عليها قواعد RTL مركزية في الـ CSS العام.

### المطلوب للـ toolbar

الـ toolbar يجب أن يكون:

* `direction: rtl`
* `display: flex`
* `flex-direction: row-reverse`
* `align-items: center`
* gap مناسب
* ترتيب العناصر صحيح تلقائيًا

لكن افعل هذا فقط على class مشتركة مخصصة للـ toolbar، وليس على أي container عام قد يُستخدم في أماكن أخرى.

### المطلوب للـ filter bars

نفس الفكرة:

* direction واضح
* flex direction مناسب
* محاذاة سليمة
* الحقول والأزرار تظهر بترتيب عربي صحيح

### المطلوب للـ pagination

عالجها بشكل explicit:

* لا تعتمد على `justify-between` فقط
* أنشئ نمط RTL صحيح للـ pagination containers
* تأكد أن:

  * نص عدد النتائج
  * أزرار السابق/التالي
  * page controls
    تظهر بترتيب صحيح في العربية

### المطلوب للـ grid / table wrappers

أضف دعم RTL على الـ wrappers الخاصة بالجداول:

* `direction: rtl` على wrapper مناسب
* وإذا لزم، `dir="rtl"` على `<table>` أو المكوّن المشترك الذي يبني الجدول
* تأكد أن رؤوس الأعمدة وصفوف البيانات تظهر بنفس النسق العربي

---

## 3) لا تعتمد على `dir="rtl"` وحده في flex layouts

افهم أن `dir="rtl"` في هذا الكود لا يعكس ترتيب flex تلقائيًا بالشكل المطلوب في كل الحالات.

لذلك:

* استخدم `dir="rtl"` للنص والسياق العام
* واستخدم `flex-row-reverse` أو `flex-direction: row-reverse` للـ shared flex containers مثل toolbars وpagination حيث يلزم

لكن:

* لا تعمم `row-reverse` على عناصر عامة جدًا
* استخدمها فقط في classes مقصودة ومشتركة

---

## 4) أصلح من الجذر وليس patch لكل شاشة

إذا وجدت شاشات قديمة بها إصلاحات RTL محلية متفرقة:

* لا تكرر نفس الإصلاح في عشرات الملفات
* حاول أولًا حلها عبر:

  * layout الأساسي
  * shared CSS classes
  * shared wrappers/components

ثم:

* أصلح فقط الحالات الاستثنائية التي لا يمكن تغطيتها من الجذر

الهدف:

* أقل تغييرات محلية ممكنة
* أكبر تغطية system-wide ممكنة

---

## 5) راجع أثر التغييرات على الصفحات التي تستخدم `flex-row-reverse` بالفعل

قبل تعميم أي rule، افحص الصفحات أو المكونات التي قد تكون أصلًا مقلوبة يدويًا.

المطلوب:

* منع double reverse
* منع انقلاب layout الصحيح
* لو فيه component أصلًا مضبوط محليًا، لا تفسده بالـ global rule

طبق الحل بذكاء وبأقل كسر ممكن.

---

## 6) أنشئ نمطًا موحدًا لأي شاشة جديدة

بعد إنهاء التعديل، يجب أن يصبح النمط كالتالي:

### أي شاشة جديدة تستخدم:

* AppLayout
* shared toolbar class
* shared filters class
* shared grid wrapper
* shared pagination class

ترث RTL الصحيح تلقائيًا بدون أي إصلاح إضافي.

إذا لزم:

* أنشئ أو حسّن shared utility classes واضحة الأسماء
* واجعل استخدامها هو النمط الرسمي المعتمد للمشروع

---

## 7) راجع المكونات الحساسة التالية بعد التعديل

اختبر وراجع بصريًا ووظيفيًا على الأقل:

* toolbars
* search bars
* filter rows
* action buttons مع icons
* table headers
* data rows
* pagination
* tabs لو متأثرة
* dialogs إن تأثرت بالاتجاه
* back buttons / navigation strips

---

## 8) لا تكسر الحالات الخاصة

بعض العناصر قد تحتاج LTR مثل:

* أرقام
* أكواد
* IDs
* email
* URLs
* حقول تقنية
* بعض الجداول ذات الطبيعة الإنجليزية

لا تكسر هذه الحالات.
اجعل RTL هو default العام، لكن حافظ على إمكانية وجود استثناءات موضعية واضحة.

---

## 9) Refactor مطلوب عند اللزوم

إذا وجدت:

* تكرار RTL hacks
* classes متضاربة
* wrappers مكررة
* CSS متعارض
* أكواد ميتة تخص محاولات RTL قديمة

فنظّفها من الجذور.

مطلوب:

* refactor نظيف
* shared pattern واضح
* إزالة الحلول المتناثرة غير الضرورية
* الحفاظ على التوافق وعدم كسر القائم

---

## 10) معايير التنفيذ

قبل التعديل:

* راجع الـ root layout
* راجع global CSS
* راجع shared classes المستخدمة عبر النظام

أثناء التنفيذ:

* نفذ الحل من الملف/الملفات الأساسية أولًا
* ثم أصلح فقط ما يحتاج استثناء
* راقب أي أثر جانبي

بعد التنفيذ:
قدّم تقريرًا واضحًا يشمل:

1. ما الذي تم تغييره في layout الأساسي
2. ما الذي تم تغييره في global/shared CSS
3. ما الـ shared classes التي أصبحت المصدر الرسمي للـ RTL
4. ما الصفحات أو المكونات التي احتاجت استثناءات
5. ما الذي تم تنظيفه أو حذفه
6. checklist للشاشات التي يجب مراجعتها يدويًا

---

# معايير القبول النهائية

لا يعتبر العمل منتهيًا إلا إذا تحقق الآتي:

* RTL مثبت من الجذر بشكل صريح
* toolbars تتصرف صح system-wide
* filter rows تتصرف صح system-wide
* grids/tables تتصرف صح system-wide
* pagination تتصرف صح system-wide
* أي شاشة جديدة ترث نفس النمط بدون ترقيع
* لا يوجد كسر واضح في الشاشات الحالية
* لا يوجد double reverse
* لا يوجد اعتماد على confirmات بصرية أو سلوكيات عشوائية
* الكود النهائي نظيف وقابل للصيانة وليس مجموعة patches متناثرة

# توجيه تنفيذي أخير

نفّذ الحل كـ **RTL infrastructure** على مستوى المشروع، وليس كإصلاح محلي لصفحة واحدة.
ابدأ من:

* root layout
* global CSS
* shared classes

ثم أصلح فقط الحالات الاستثنائية.
لا تكرر المنطق.
لا تكسر أي شاشة قائمة.
اجعل أي شاشة جديدة أو قديمة تعمل بنفس النمط تلقائيًا.
