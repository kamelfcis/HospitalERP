أريدك أن تنفّذ مراجعة وتحسين قوية ومحترفة على موديول الخدمات + قوائم الأسعار + تسعير التعاقدات + فاتورة المريض داخل النظام الحالي، لكن بشرط صارم جدًا:

- ابنِ على الموجود حاليًا ولا تهدم أي جزء شغال.
- لا تعمل إعادة اختراع للنظام.
- لا تكسر أي flow قائم.
- أي تعديل لازم يكون backward-compatible قدر الإمكان.
- اعمل Refactor محترم، Componentization واضح، وفهرسة مناسبة.
- اعتمد دائمًا على أقل تغيير يحقق أعلى أثر.
- لا تنقل منطق حساس إلى الواجهة إذا كان مكانه الصحيح في الخادم.
- اجعل السيرفر هو authoritative source في كل ما يخص التسعير النهائي.
- نفّذ الحلول الأنسب بناءً على مسار السيستم الحالي والكود الموجود فعلاً، وليس بناءً على افتراضات نظرية.

## المطلوب منك أولًا — تحليل دقيق قبل التنفيذ
ابدأ بتحليل الوضع الحالي فعليًا في الكود والجداول والعلاقات ومسار الفاتورة والتسعير، ثم قرر الحل الأنسب بناءً على الموجود حاليًا. لا تنفذ بعشوائية.

راجع خصوصًا هذه النقاط المهمة:

1) الخدمات مربوطة بالفعل بالأقسام، لكن ابحث هل فلترة الخدمات داخل فاتورة المريض مفعلة فعلاً أم يوجد prop / filter ناقص في الشاشة أو الكمبوننت أو الـ API.

2) راجع قوائم الأسعار الحالية:
- هل يوجد price_list_type
- هل يوجد department_id
- هل يوجد valid_from / valid_to / is_active
- هل يوجد default concept أم لا
- وهل فكرة default الحالية لو أُضيفت يجب أن تكون:
  - على مستوى النظام كله
  - أم على مستوى النوع
  - أم على مستوى القسم
  - أم على مستوى النوع + القسم
اختر الأنسب بناءً على تصميم النظام الحالي حتى لا نصنع تعارضات مستقبلية.

3) راجع العقود:
- هل contracts.base_price_list_id موجود فقط في schema أم مستخدم فعلاً
- هل يتم استدعاؤه داخل منطق التسعير أم لا
- هل محرك التغطية يقرأ list price فعلاً من price_list_items أم يبدأ من base_price فقط
- إن كان غير مستخدم، فعّل استخدامه بالطريقة الصحيحة الأقل هدمًا والأكثر اتساقًا مع الكود الحالي.

4) راجع منطق التسعير بين الواجهة والخادم:
- هل الواجهة تعرض سعرًا من services.base_price فقط
- وهل الخادم يعيد حساب السعر عند الحفظ/الفاينال
- وهل هناك احتمال اختلاف بين السعر المعروض والسعر النهائي
إن وُجد هذا التضارب، عالجه من الجذر بالحل الأنسب معماريًا بناءً على الموجود.

5) راجع صلاحية قوائم الأسعار:
- active
- validity dates
- نوع القائمة
- ارتباطها بالقسم أو العقد
- وجود الخدمة داخل القائمة
ولا تسمح باستخدام قائمة أسعار غير صالحة أو منتهية أو غير مناسبة للسياق.

6) راجع منطق الـ fallback:
إذا لم توجد الخدمة في قائمة أسعار العقد، أو القائمة الافتراضية، أو كانت غير صالحة:
- هل الأفضل fallback آمن
- أم warning واضح
- أم block
اختر السلوك الأنسب حسب منطق النظام الحالي، لكن امنع أي fallback صامت يسبب تسعيرًا غير مفهوم أو غير قابل للمراجعة.

7) راجع فكرة التعديل اليدوي على السعر:
خصوصًا في حالة مريض تعاقد أو وجود قائمة أسعار.
افحص هل النظام الحالي يسمح manual override، وهل هذا مناسب، وهل يحتاج permission أو audit أو lock.
لا تضف تعقيدًا غير لازم، لكن إن كانت هناك ثغرة رقابية واضحة فاقفلها بالأسلوب الأنسب.

8) راجع خصائص الخدمة:
- requires_doctor
- requires_nurse
- service_type
- category
- service_consumables
ولا تهدم الموجود.
ولو ستضيف نوعًا جديدًا مثل NURSING فافعل ذلك فقط إذا كان متسقًا فعلاً مع النظام الحالي وله قيمة تشغيلية واضحة الآن، وليس لمجرد الكمال النظري.

## المطلوب منك ثانيًا — قرارات تصميم عملية based on existing system
بعد التحليل، اتخذ القرارات الصحيحة بنفسك بناءً على الكود الحالي، مع مراعاة ما يلي:

- قدم أهم إصلاحات التأثير العالي أولاً.
- لا تبدأ بالكماليات قبل علاج الفجوات المؤثرة على صحة التسعير وسلامة التشغيل.
- لو كانت هناك نقطة بسيطة جدًا لكن أثرها كبير (مثل filter ناقص أو prop غير ممرر) فابدأ بها فورًا.
- لو كانت هناك نقطة معمارية حساسة تمس صحة الفاتورة أو التسعير النهائي، فاجعلها أعلى أولوية من التحسينات الشكلية.

## المطلوب منك ثالثًا — تنفيذ نظيف ومعماري
نفّذ الحل مع الالتزام الكامل بالآتي:

### A) Refactor
- لا تكدّس المنطق داخل route أو component ضخم.
- افصل المنطق في services / helpers / resolvers / hooks / storage حسب مكانه الصحيح.
- اجعل أي pricing resolution logic في نقطة مركزية واحدة يمكن الاعتماد عليها.
- امنع تكرار منطق التسعير في أكثر من مكان.

### B) Componentization
- أي UI logic متكرر افصله في component أو hook واضح.
- اجعل ServiceLookup / price selection / contract pricing preview / invoice line pricing أكثر نظافة وقابلية لإعادة الاستخدام.
- لا تضع business rules الثقيلة داخل JSX.

### C) Server Authority
- السعر النهائي، list price، contract adjustments، fallback logic، validity checks، كل هذا يجب أن يكون محسومًا على الخادم.
- الواجهة يمكنها عرض preview أو prefill فقط.
- عند الحفظ أو finalize يجب أن يعاد الحساب من السيرفر بشكل authoritative.

### D) Auditability
أضف trace / audit مناسب بدون مبالغة:
- مصدر السعر النهائي
- هل جاء من contract price list أو default price list أو service base price
- price list id المستخدم
- contract rule id إن وُجد
- نوع الـ fallback إن حصل
- أي manual override إن كان مسموحًا
بحيث نستطيع مراجعة سبب السعر لاحقًا.

### E) Backward Compatibility
- لا تكسر الفواتير القديمة.
- لا تكسر التقارير القديمة.
- لا تغيّر semantics أساسية إلا لو كان هناك خطأ واضح.
- أي migration أو schema change لازم تكون minimal وآمنة.
- لا تفرض حقولًا جديدة على بيانات قديمة بدون handling واضح.

### F) Indexing & Performance
راجع الفهارس اللازمة بعناية، خاصة على:
- services.department_id
- services.service_type لو عليه filtering فعلي
- price_lists (is_active / validity / type / department)
- price_list_items (price_list_id, service_id)
- contracts.base_price_list_id
- أي استعلامات تستخدم في البحث أو تسعير الفاتورة
أضف فقط الفهارس المفيدة فعلاً، وتجنب الفهارس العشوائية.

### G) Safety
- لا تسمح بتسعير غير واضح.
- لا تسمح بتضارب بين سعر الواجهة وسعر الحفظ النهائي.
- لا تسمح باستخدام قائمة أسعار غير صالحة بدون تسجيل ذلك بوضوح.
- لا تكسر الأداء داخل شاشة الفاتورة.
- لا تضف N+1 queries داخل مسار حساس.

## المطلوب منك رابعًا — ترتيب التنفيذ
نفّذ على مراحل منطقية. أنت تحدد الأولوية الأنسب بناءً على الكود الحالي، لكن راعِ أن تكون الأولوية عادة كالتالي إن كانت متسقة مع الواقع:
- إصلاح الفلترة الناقصة أو الربط الناقص في شاشة فاتورة المريض
- توحيد/تصحيح منطق التسعير النهائي
- تفعيل قائمة أسعار العقد إن كانت موجودة وغير مستخدمة
- منع التضارب بين UI preview و server pricing
- ثم التحسينات الثانوية مثل default list strategy أو type expansion أو تحسينات UX

لكن لا تلتزم حرفيًا بهذا الترتيب إذا كشف التحليل أن النظام الحالي يحتاج ترتيبًا مختلفًا. اختر الأنسب فعلاً.

## المطلوب منك خامسًا — مخرجات التنفيذ
أريد منك في الرد النهائي:

1) Current State الحقيقي بعد مراجعة الكود
2) الفجوات الفعلية فقط
3) القرار المعماري الذي اخترته ولماذا هو الأنسب للنظام الحالي
4) خطة تنفيذ مختصرة مرتبة
5) التنفيذ الكامل
6) أي schema changes أو indexes تمت إضافتها
7) ماذا حافظت عليه حتى لا تكسر النظام
8) ماذا أصلحت في التسعير والفلترة والربط
9) Done looks like
10) أي مخاطر أو نقاط مؤجلة إن وجدت

## تنبيهات صارمة
- لا تهدم.
- لا توسّع النطاق بلا داعٍ.
- لا تضف تعقيدًا غير مبرر.
- لا تكرر منطق التسعير في أكثر من طبقة.
- لا تجعل الواجهة هي مصدر الحقيقة.
- لا تنفذ حلولًا تجميلية وتترك أصل المشكلة.
- لا تكتفِ بإضافة حقول schema غير مستخدمة.
- أي field موجود بالفعل وغير مستخدم، افحص هل الأفضل تفعيله بدل اختراع field جديد.
- اشتغل بعقلية enterprise ERP محترمة: دقة، traceability، performance، backward compatibility، auditability.

نفّذ الآن بعد أن تفحص الموجود فعليًا، واختر أنت الحلول الأنسب حسب بنية النظام الحالية.
