SYSTEM TASK
Implement a secure, backend-enforced Patient Inquiry screen for the hospital ERP system.

This feature must allow users to search and view patient financial/service activity (services, drugs, consumables) while enforcing strict department-level isolation at the backend level.

Frontend filtering is ONLY for UX. Security must be enforced entirely in the backend.

The implementation must follow the architecture and constraints below exactly.

------------------------------------------------

OBJECTIVE

Build a Patient Inquiry module that displays:

• Patient identity
• Patient phone
• Department
• Total services
• Total drugs
• Total consumables
• Total net invoices
• Total paid
• Outstanding balance
• Last invoice date

And allow opening a detailed side panel showing the lines grouped by:

• Services
• Drugs
• Consumables

Data source is the existing invoice system:

patient_invoice_headers  
patient_invoice_lines

------------------------------------------------

CRITICAL SECURITY RULES

R1  
Non-admin users must ONLY see invoices belonging to their department.

Always enforce:

WHERE pih.department_id = req.user.departmentId

R2  
Non-admin users must NEVER control department filtering through query parameters.

Ignore any deptId passed in the URL for non-admin users.

Use only:

forcedDeptId = req.user.departmentId

R3  
Invoices with department_id IS NULL must be hidden from non-admin users.

R4  
Patient matching priority:

1️⃣ patient_id  
2️⃣ fallback to patient_name ONLY if patient_id is NULL

R5  
Admissions data must NOT be shown to non-admin users in Phase 1.

R6  
If a non-admin user has no department assigned → return HTTP 403.

R7  
Clinic filtering is only allowed when the effective department is OPD.

R8  
All totals must be calculated WITHOUT duplication caused by JOINing invoice lines.

R9  
Lines endpoint must verify that the requested patient actually has invoices inside the user's department scope.

R10  
API responses must return metadata:

count  
limit  
hasMore

R11  
Date filters must be inclusive of the full day (avoid time boundary errors).

R12  
Every access to Patient Inquiry endpoints must be audit logged.

T001
BACKEND STORAGE FUNCTIONS

File

server/storage/patients-doctors-storage.ts

Create two functions.

------------------------------------------------

FUNCTION 1

getPatientInquiry(filters, forcedDeptId, isAdmin)

IMPORTANT

Do NOT directly join invoice headers and lines when calculating totals.

This causes header totals to multiply by number of lines.

Instead use CTE structure.

Required SQL structure:

CTE 1
filtered_invoices

Select only invoices allowed for the user.

Apply rules:

• status != cancelled
• department isolation
• date filters
• clinic filter (if applicable)

CTE 2
invoice_totals

Aggregate header totals per patient.

SUM net_amount  
SUM paid_amount  
SUM outstanding

CTE 3
line_totals

Aggregate totals by line_type

SUM services  
SUM drugs  
SUM consumables

CTE 4
final join

Join the results.

Fields returned:

uid  
patient_id  
patient_code  
patient_name  
patient_phone  
department_id  
dept_name  
invoice_count  
services_total  
drugs_total  
consumables_total  
total_net  
total_paid  
total_outstanding  
last_invoice_date

Sort:

ORDER BY last_invoice_date DESC

Limit:

LIMIT 200

Return metadata:

count  
limit  
hasMore
