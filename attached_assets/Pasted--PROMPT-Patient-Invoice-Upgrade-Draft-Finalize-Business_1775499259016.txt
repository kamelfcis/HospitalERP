🧠 PROMPT — Patient Invoice Upgrade (Draft/Finalize + Business Classification + Enterprise Refactor)

أريد تنفيذ ترقية احترافية وآمنة لنظام فاتورة المريض، بناءً على النظام الحالي، مع الالتزام الكامل بعدم كسر أي منطق قائم، خصوصًا:

🎯 الهدف العام

تحقيق 3 أهداف رئيسية:

1) نموذج حالات الفاتورة

تحويل الفاتورة إلى نظام:

Draft (مسودة) → مرنة بالكامل
Finalized (معتمدة) → مغلقة تمامًا
2) إضافة Business Classification

إضافة تصنيف موحد لكل بند داخل الفاتورة (أدوية / خدمات / تحاليل / أشعة / إقامة / غازات… إلخ)

3) تنفيذ Refactor احترافي
بدون كسر النظام
مع تحسين الأداء
مع استخدام compound design
مع فهرسة ذكية

1️⃣ Invoice State Model (Draft / Finalized)
المطلوب

إضافة حالة للفاتورة:

status: "draft" | "finalized"
السلوك
Draft:
إضافة / تعديل / حذف البنود
تعديل الأسعار والخصومات
تعديل المريض
تعديل business_classification
Finalized:
❌ لا تعديل
❌ لا حذف
❌ لا إضافة
❌ لا تغيير business_classification
Backend Rules
عند التعديل:
if (invoice.status === "finalized") {
  throw HTTP 422 "INVOICE_FINALIZED_NO_EDIT"
}
خاص بـ business_classification:
مسموح في Draft
ممنوع بعد Finalized
Frontend
زر: حفظ مسودة
زر: اعتماد الفاتورة
2️⃣ Business Classification Design
التصنيفات
enum BusinessClassification {
  drug
  consumable
  equipment
  gas
  medical_service
  lab
  radiology
  accommodation
  operating_room
}
أماكن التخزين
items.businessClassification
services.businessClassification
patient_invoice_lines.business_classification

⚠️ جميعها nullable

مهم جدًا
التصنيف يأتي من master data
الفاتورة فقط تخزن snapshot
لا يعتمد على UI
3️⃣ Central Resolver (مهم جدًا)

ملف واحد فقط:

resolveBusinessClassification()
القواعد:
Stay Engine:
if (sourceType === "STAY_ENGINE") return "accommodation"

يستخدم service.businessClassification
fallback على serviceType
Item:
يستخدم item.businessClassification
fallback على lineType
⚠️ Fallback Logging
