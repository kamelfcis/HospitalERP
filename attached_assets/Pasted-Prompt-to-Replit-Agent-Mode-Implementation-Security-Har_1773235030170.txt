Mode: Implementation + Security Hardening
Objective:
Implement a Patient Inquiry Screen that allows users to query patient activity (services, drugs, consumables) while strictly enforcing department-level data isolation.

The system must guarantee that:

Admins can see all departments

Non-admin users can ONLY see their own department

No user can bypass this restriction via query parameters or URL manipulation

Security enforcement must be server-side, not frontend-only.

Critical Security Rules (Must Be Enforced)
Rule 1 — Department Isolation

For non-admin users:

ALL queries must enforce:
pih.department_id = req.user.departmentId

Even if a user sends:

/api/patient-inquiry?deptId=anotherDept

The backend must ignore it.

Department restriction must always come from:

req.user.departmentId
Rule 2 — Ignore Query Department for Non-Admins

If:

user.role != 'admin'

Then:

deptId from query params MUST be ignored

Only admins may filter departments manually.

Rule 3 — Prevent Data Leakage from NULL Departments

Invoices where:

department_id IS NULL

must NOT appear to non-admin users.

Admins may see them.

Rule 4 — Secure Patient Matching

Avoid using patient name as primary identifier.

Logic must be:

IF patient_id exists → use patient_id
ELSE fallback to patient_name (legacy records only)

Fallback using patient name must still enforce department restriction.

Rule 5 — Admissions Data Visibility

Admissions currently have no department_id.

To prevent cross-department data leakage:

Phase 1 rule:

Admissions should NOT be included in Patient Inquiry results
for non-admin users unless a reliable department mapping exists.

Admins may see them.

Rule 6 — API Authorization

Access to this screen requires:

patients.view permission

Additionally:

if user.role != admin AND user.department_id IS NULL
→ return HTTP 403

Do NOT rely on frontend validation.
