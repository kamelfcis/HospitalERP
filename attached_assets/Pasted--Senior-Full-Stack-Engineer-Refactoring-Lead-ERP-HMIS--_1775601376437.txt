# TASK #40 — إصلاح سطور الإقامة في فاتورة المريض

## الهدف

إصلاح منطق سطور الإقامة (Stay Engine) في الخلفية والواجهة بحيث تصبح:

* صحيحة محاسبيًا وتشغيليًا
* موحّدة معماريًا
* سهلة الصيانة
* وتُحدَّث لحظيًا في فاتورة المريض بدون سلوك مزعج في الإدخال

---

# السياق الحالي

محرك الإقامة يعمل جزئيًا:

* عند قبول المريض في سرير: ينزل سطر إقامة فورًا
* عند تحويل المريض بين الغرف: ينزل سطر جديد فورًا
* لكن يوجد 5 بجات حقيقية في الخلفية والواجهة تؤثر على:

  * صحة البيانات
  * business_classification
  * quantity / total_price
  * تجربة الإداري في تعديل الكمية
  * قابلية صيانة الكود

---

# Done looks like

يُعتبر العمل منتهيًا فقط إذا تحقق الآتي بالكامل:

1. عند قبول مريض في سرير:

   * ينزل سطر إقامة فورًا
   * quantity = 1
   * السعر = سعر الغرفة
   * `business_classification = 'accommodation'`

2. عند تحويل المريض:

   * ينزل سطر جديد فورًا
   * بسعر الغرفة الجديدة
   * `business_classification = 'accommodation'`

3. عند التراكم اليومي `accrueStayLines`:

   * كل سطر جديد ينزل بـ quantity = 1
   * `total_price` صحيح ومساوٍ للسعر
   * لا توجد NULL أو zero totals الخاطئة

4. الإداري يمكنه تعديل الكمية بحرية:

   * 0.5
   * 1.5
   * أي قيمة عشرية صحيحة
   * بدون قيود غير منطقية

5. الفاتورة تتحدث لحظيًا بعد التعديل

6. كود إدراج سطر الإقامة يصبح موحّدًا:

   * helper واحد مشترك
   * لا تكرار 3 نسخ من نفس المنطق

---

# Out of scope

ممنوع التوسع خارج هذه المهمة. لا تلمس:

* FEFO
* منطق الأدوية
* صلاحيات جديدة لسطور الإقامة
* طريقة احتساب الأيام (`hours_24` / `hotel_noon`)
* partial unique index (تم سابقًا)

---

# البجات المؤكدة التي يجب علاجها

## أولًا: بجات الخلفية

1. `accrueStayLines` يُدرج:

* `quantity = NULL`
* `total_price = '0'`

وهذا خطأ. يجب إصلاحه ليصبح:

* `quantity = '1'`
* `total_price = rateStr`

2. `admitPatientToBed` يفتقد:

* `business_classification = 'accommodation'`

3. `transferPatientBed` يفتقد:

* `business_classification = 'accommodation'`

4. يوجد تكرار كود إدراج stay line في 3 أماكن:

* `admitPatientToBed`
* `transferPatientBed`
* `accrueStayLines`

هذا يجب إنهاؤه باستخراج helper مركزي واحد.

---

## ثانيًا: بجات الواجهة

1. حقل الكمية يمنع الكسور بسبب:

* `min=1`
* `step="1"`

وهذا يجب تغييره بحيث يقبل:

* `min="0"`
* `step="any"`

2. الـ input controlled بشكل يسبب:

* رجوع مؤشر الكتابة
* تجربة سيئة أثناء التعديل

يجب تحويله إلى:

* uncontrolled input
* `defaultValue`
* `key` reset عند الحاجة
* بدون تهييج للمؤشر

3. `isAutoGenerated` يعتمد حاليًا على:

* `priceSource === "stay_engine"`

وهذا لا يعمل لأن القيمة الحقيقية القادمة من قاعدة البيانات هي:

* `source_type = 'STAY_ENGINE'`

إذن يجب تصحيح المنطق ليعتمد على:

* `sourceType === 'STAY_ENGINE'`
  فقط، وبوضوح.

# المطلوب تنفيذه

## المرحلة 1: تحليل سريع ودقيق قبل التعديل

راجع الملفات التالية أولًا:

* `server/storage/bedboard-stays-storage.ts`
* `server/storage/bedboard-beds-storage.ts`
* `client/src/pages/patient-invoice/components/UnifiedLinesTab.tsx`

حدّد بدقة:

* أماكن الإدراج الثلاثة الحالية
* شكل البارامترات الحالية
* الفروقات بينها
* أين يتم حساب `quantity`
* أين يتم حساب `total_price`
* أين يغيب `business_classification`
* كيف يتم تمييز سطر `STAY_ENGINE` في الواجهة حاليًا

ثم نفّذ.

---

## المرحلة 2: استخراج helper مشترك للإدراج

أنشئ helper مشترك نظيف، باسم واضح مثل:

```ts
buildStayLineParams(options)
```

أو اسم أدق إن لزم، بشرط أن يكون:

* واضحًا
* قابلًا لإعادة الاستخدام
* في مكان منطقي مشترك بين:

  * `admitPatientToBed`
  * `transferPatientBed`
  * `accrueStayLines`

## هذا الـ helper يجب أن يبني سطر STAY_ENGINE بشكل موحّد ويشمل:

* source_type الصحيح
* quantity
* unit_price / rate
* total_price المحسوبة
* `business_classification = 'accommodation'`
* أي fields لازمة مشتركة
* بدون تكرار منطق مشتت

## المطلوب المعماري

* لا تترك 3 نسخ متشابهة من INSERT
* اجعل الإدراج يمر من نفس المصدر المنطقي
* إن كان هناك اختلاف محدود بين السيناريوهات، مرره كـ options واضحة

---

## المرحلة 3: إصلاح `accrueStayLines`

استبدل الإدراج الحالي الخاطئ بحيث يصبح:

* `quantity = '1'`
* `total_price = rateStr`

وباستخدام الـ helper المشترك، وليس patch يدوي منفصل.

تأكد أن:

* لا تُنتج أسطر بإجمالي صفر
* لا تُنتج أسطر quantity فارغة
* السطر الناتج يطابق منطق أول يوم والتحويل من حيث classification والبناء

---

## المرحلة 4: إصلاح `admitPatientToBed` و `transferPatientBed`

حوّل منطق الإدراج فيهما لاستخدام الـ helper المشترك أيضًا.

وتأكد أن كلاهما يضيف:

* `business_classification = 'accommodation'`

ممنوع ترك واحد منهم خارج التوحيد.

---

## المرحلة 5: إصلاح الواجهة في `UnifiedLinesTab`

المطلوب هنا دقيق جدًا:

### أ) إصلاح حقل الكمية

حوّل حقل الكمية الخاص بسطور STAY_ENGINE من:

* controlled input

إلى:

* uncontrolled input
* `defaultValue`
* reset ذكي باستخدام `key` عند الحاجة

بحيث:

* الكتابة تكون سلسة
* المؤشر لا يرجع
* التعديل لا يتهيج أثناء typing

### ب) دعم الكسور

استبدل:

* `min=1`
* `step="1"`

بـ:

* `min="0"`
* `step="any"`

بحيث يقبل:

* 0.5
* 1.5
* 2.25
* إلخ

### ج) إصلاح auto-generated logic

نظّف منطق:

* `isAutoGenerated`

بحيث يعتمد على:

* `sourceType === 'STAY_ENGINE'`

ولا يعتمد على:

* `priceSource === "stay_engine"`

إلا إذا كان هناك سبب حقيقي موثّق لاستخدامه، وإلا احذفه من هذا الفرع.

---

## المرحلة 6: التحديث اللحظي بعد التعديل

تأكد أن تعديل الكمية في الواجهة:

* ينعكس فورًا على الفاتورة
* يحدّث الإجمالي
* لا يحتاج refresh يدوي
* لا يكسر باقي أنواع السطور

إذا كان هناك cache invalidation / local state sync / mutation update مطلوبة:

* نفّذها بشكل نظيف
* بدون hack
* وبدون كسر الأداء

# متطلبات هندسية صارمة

## 1) Refactor حقيقي

ممنوع patch سريع.
مطلوب:

* helper مشترك
* إزالة التكرار
* naming واضح
* فصل المنطق المشترك عن السيناريوهات

## 2) لا تكسر السلوك الحالي الصحيح

حافظ على:

* نزول سطر أول يوم فورًا
* نزول سطر التحويل فورًا
* التراكم اليومي
* شكل الفاتورة الحالي
* باقي أنواع السطور

## 3) لا تخلط بين منطق الإقامة وباقي البنود

أي منطق خاص بـ:

* drugs
* consumables
* services
* equipment

لا تلمسه إلا لو كان هناك coupling لازم جدًا، وحينها بأقل قدر ممكن.

## 4) Naming واضح

استخدم أسماء تعبّر بدقة عن:

* stay line params
* source type
* accommodation classification
* generated stay line

## 5) Safety

* لا تغيّر schema بلا داعٍ إن لم تكن المهمة تتطلبه
* لا تلمس index أو migrations خارج الحاجة
* لا توسع نطاق المهمة
