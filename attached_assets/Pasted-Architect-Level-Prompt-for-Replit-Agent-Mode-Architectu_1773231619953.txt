System Context

This system currently includes modules for:

Patients

Clinic appointments

Consultations

Admissions / stays

Invoices

Payments

However, the patient file is incomplete and invoices currently reference the patient name instead of patient_id, which is unsafe.

The objective is to transform the patient profile into a complete longitudinal patient record.

Core Architectural Goals

Establish patient_id as the single source of truth across financial and clinical records.

Introduce a unified patient timeline showing the full journey of the patient.

Maintain data integrity and auditability.

Avoid breaking existing system behavior.

Ensure performance scalability for large datasets.

Required Architectural Improvements
1️⃣ Safe Migration to patient_id

Invoices currently link patients using patient_name.

Design a migration that:

Adds patient_id to patient_invoice_headers

Populates it safely using matching logic.

Rules:

If exactly one patient matches the invoice name → link

If multiple matches → leave NULL

If no match → leave NULL

Produce a migration audit report listing:

matched invoices

ambiguous matches

unmatched invoices

Also include a database backup step before migration.

2️⃣ Establish Patient Encounter Layer

Introduce a clinical encounter model to structure the patient journey.

Create a new table concept:

patient_encounters

Fields:

id
patient_id
encounter_type (clinic / emergency / admission)
doctor_id
department_id
start_time
end_time
status
created_at

All clinical and billing events should optionally reference:

encounter_id

This allows grouping services under a visit.
