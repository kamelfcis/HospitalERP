أريد تنفيذ ترقية آمنة ومدروسة على النظام الحالي لترسيخ قاعدة أساسية وهي:

كل مريض داخل المستشفى يجب أن يكون له كود موحد ثابت يربط كل الخدمات الطبية والمالية والإدارية التي حصل عليها من أول الدخول أو الكشف وحتى المغادرة.

لكن مهم جدًا أن يكون التنفيذ Upgrade على الموجود الحالي وليس إعادة تصميم تكسر النظام أو تغيره جذريًا.

الهدف الرئيسي

مراجعة النظام الحالي كما هو موجود الآن، ثم:

اكتشاف هل يوجد بالفعل Patient Master / ملف مريض / patient_id / patient code موحد
تقييم مدى استخدامه فعليًا في جميع الشاشات والمعاملات
استكمال أي نقص موجود
توحيد الربط بين الخدمات المختلفة على كود المريض الموحد
تنفيذ ذلك بدون كسر النظام الحالي أو تعطيل الشاشات أو فقد البيانات
أي تعديل يكون Refactor محسوب وآمن وبأقل تغيير ممكن في السلوك الحالي
قواعد التنفيذ الإلزامية
1) اعمل Audit أولًا على الموجود الحالي

قبل أي تعديل، راجع النظام الحالي بالكامل وحدد بوضوح:

هل يوجد جدول مرضى رئيسي؟
هل يوجد patient_id موحد؟
هل توجد شاشات تستخدم اسم المريض كنص فقط بدل الربط الفعلي؟
ما هي الجداول التي ترتبط بالمريض حاليًا؟
ما هي الجداول أو الشاشات التي ينقصها الربط؟
هل يوجد Visit / Encounter / Admission / Episode حاليًا؟
ما الفرق بين الموجود حاليًا وبين التصميم المطلوب؟

ممنوع افتراض أن النظام ناقص بالكامل.
ابدأ دائمًا من الموجود وكمّل عليه.

2) ممنوع كسر النظام الحالي

أي تعديل يجب أن يلتزم بالآتي:

عدم حذف أي أعمدة أو جداول مستخدمة حاليًا مباشرة
عدم تغيير أسماء أعمدة أو API endpoints أو props أو contracts بشكل يكسر الشاشات
عدم إلغاء أي behavior قائم إلا إذا كان خطأ واضح
لو هناك حقول نصية حالية مثل patient_name، لا تُحذف فورًا، بل:
تبقى كـ snapshot / legacy / display field عند الحاجة
ويتم الاعتماد تدريجيًا على patient_id كمرجع أساسي
أي خطوة Migration تكون تدريجية وآمنة وقابلة للرجوع
3) المطلوب هو Upgrade وليس Rewrite

نفّذ الحل كالتالي:

Reuse الموجود الحالي قدر الإمكان
أضف فقط الناقص
اعمل refactor في الأماكن التي تحتاج فقط
اجعل الكود Modular / reusable / composable
لا تعيد بناء modules كاملة طالما يمكن ترقيتها جزئيًا
التصميم المطلوب منطقيًا
الكيان الأساسي

يجب أن يكون هناك Patient Master واحد فقط لكل مريض:

patients.id
patients.patient_code أو mrn
بيانات المريض الأساسية
ثم الارتباط بالمستويات التشغيلية إن كانت موجودة أو ينقصها استكمال:
زيارة encounter / visit
دخول admission
فاتورة
خدمة
دواء
معمل
أشعة
عملية
إقامة
مطالبة
تحصيل
قيد محاسبي مرجعي عند الحاجة
المطلوب من المبرمج عمليًا
أولًا: تحليل الحالي

أخرج تقرير تنفيذي واضح يوضح:

ما الموجود بالفعل في النظام
ما الناقص
ما الذي سيُضاف
ما الذي سيظل كما هو
ما الذي سيتم refactor له فقط
كيف سيتم التنفيذ بدون breaking changes
ثانيًا: استكمال الربط

أي جدول أو شاشة أو معاملة مرتبطة بالمريض يجب مراجعتها:

لو فيها patient_id بالفعل → تأكد أنه المستخدم فعليًا بشكل صحيح
لو فيها اسم مريض فقط أو ربط ناقص → أضف الربط الصحيح تدريجيًا
لو هناك شاشة تُنشئ مريض كنص فقط → عدّلها بحيث تنشئ أو تختار ملف مريض فعلي
ثالثًا: منع التكرار

يجب أن يضمن النظام أن نفس المريض لا يتكرر بدون داعٍ، وذلك عبر:

البحث أولًا قبل الإنشاء
استخدام مفاتيح مثل:
الرقم القومي إن وجد
رقم الموبايل
الاسم + تاريخ الميلاد عند الحاجة
ثم استخدام نفس ملف المريض القائم إن كان موجودًا

لكن لا يتم فرض منطق قاسٍ يوقف الشغل الفعلي بدون مبرر.
نريد smart deduplication وليس تعطيل الاستقبال.

قواعد مهمة جدًا للشاشات
من أي شاشة يتم التعامل مع المريض:

مثل:

الاستقبال
الطوارئ
العيادات
الإقامة
الصيدلية
الفواتير
الكاشير
الخدمات

يجب أن يكون السلوك كالتالي:

اختيار مريض موجود
أو إنشاء مريض جديد بشكل صحيح داخل master
ثم حفظ patient_id الحقيقي
وليس مجرد اسم نصي فقط

ولو كان النظام الحالي يعتمد على نص في بعض الأماكن، نفّذ compatibility layer مؤقتة إلى أن يكتمل الربط.

Backward Compatibility

مهم جدًا:

احتفظ بالتوافق مع البيانات الحالية
اعمل migration scripts آمنة
اربط البيانات القديمة تدريجيًا بالمريض الصحيح
لا تفترض أن كل البيانات القديمة نظيفة 100%
أضف أدوات review / validation / reconciliation لو لزم
Refactor المطلوب

نفّذ refactor احترافي محسوب فقط في الأجزاء التي تحتاج ذلك، مع مراعاة:

استخراج patient resolution logic في service واحدة مركزية
توحيد lookup / create / attach logic
إزالة التكرار من الشاشات والخدمات
عدم تكرار نفس منطق البحث عن المريض في أكثر من مكان
جعل الكود composable وقابل للتوسع مستقبلًا
الفهرسة والأداء

راجع الفهارس الموجودة ثم أضف الناقص فقط.

مطلوب فهرسة مناسبة على سبيل المثال حسب الموجود الفعلي:

patients(patient_code)
patients(national_id) إذا مناسب
patients(phone) إذا مناسب
encounters(patient_id, status)
admissions(patient_id, status)
أي جداول معاملات عليها:
patient_id
encounter_id
admission_id

ولو توجد تقارير ثقيلة على مسار المريض الكامل، أضف compound indexes مناسبة بناء على الاستعلامات الفعلية، وليس نظريًا فقط.

Compound / Enterprise Considerations

التنفيذ لازم يكون enterprise-friendly:

قابل للتوسع للفروع مستقبلًا
يدعم تتبع رحلة المريض كاملة
يدعم financial traceability
يدعم medical traceability
يسمح مستقبلًا بتجميع:
كل خدمات المريض
تكلفة الحالة
إيراد الحالة
صافي نتيجة الحالة
التحليل حسب القسم / الطبيب / العقد / الزيارة
المطلوب النهائي من التنفيذ

أريد التنفيذ في صورة:

1. Analysis

تحليل الموجود الحالي وما ينقصه

2. Safe Implementation Plan

خطة تنفيذ آمنة على مراحل بدون breaking changes

3. Schema / DB Changes

فقط التعديلات اللازمة فعلًا

4. Backend Refactor

توحيد منطق ربط المريض

5. Frontend Adjustments

تعديلات بسيطة وآمنة على الشاشات الناقصة فقط

6. Migration / Backfill

ربط البيانات القديمة تدريجيًا

7. Indexing

إضافة الفهارس اللازمة فقط حسب الاستعلامات الحقيقية

8. Validation

منع إنشاء حركات جديدة بدون ربط صحيح بالمريض كلما كان ذلك ممكنًا بدون تعطيل النظام

مخرجات مهمة جدًا

في النهاية أريد:

قائمة بما كان موجودًا بالفعل
قائمة بما تمت إضافته
قائمة بما لم يتم المساس به حفاظًا على الاستقرار
توضيح أي مخاطر محتملة
والتأكد أن التنفيذ يرقي النظام ولا يكسره
قاعدة حاكمة

ابدأ من الموجود الحالي، افهمه، ثم كمّل الناقص فقط.
لا تعيد اختراع النظام من الصفر.
المطلوب ترقية آمنة، متدرجة، قابلة للصيانة، وتراعي الأداء والـ refactor والـ compound design والـ indexing بدون أي breaking changes.
