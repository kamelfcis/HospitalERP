You are a senior full-stack engineer. Build a clean, production-grade “Services & Price Lists” module for a Hospital/Pharmacy ERP. Focus on simplicity, performance, and bug-free logic. Use minimal, readable code and consistent patterns. No overengineering.

TECH
- Stack: (use the project’s existing stack and conventions)
- DB: SQL (use existing connection / ORM already in the repo)
- UI: fast searchable tables, modal forms, inline edit where appropriate
- Add server-side validation + client-side validation
- Must be performant with 10,000+ services and multiple price lists

GOAL
One module that supports:
1) Defining services (Service Master)
2) Creating multiple price lists (Cash, Insurance, Company A, etc.)
3) Assigning a price for each service per price list
4) Bulk adjustments: apply +/- % or fixed amount to a whole price list (optionally filtered by department/category)
5) Smooth UX: quick search, filters, favorites, minimal clicks, no clutter

CORE CONCEPTS
- “Service Master” is the truth: service identity + department + accounting mapping (revenue account, cost center) + optional default warehouse for filtering only.
- “Price List” contains prices per service (many price lists possible).
- Service can have a BasePrice default, but actual price used in billing comes from selected price list; if missing, fallback to BasePrice (configurable).

DATA MODEL (create tables if not exist, or migrate safely)
1) Departments
- id (PK), code (unique), name

2) Warehouses
- id (PK), code (unique), name, department_id (nullable)

3) Services (Service Master)
- id (PK)
- code (unique, indexed)
- name_ar, name_en (nullable)
- department_id (FK, indexed)  -- required
- category (string, nullable, indexed)  -- e.g., Surgery, ICU, Lab, Radiology, Accommodation, Devices, Gases
- service_type (enum/string)  -- SERVICE / ACCOMMODATION / DEVICE / GAS / OTHER
- default_warehouse_id (FK nullable) -- ONLY for filtering UX, not inventory movement
- revenue_account_id (FK/number, required)
- cost_center_id (FK/number, required)
- base_price (decimal(18,2) default 0)
- is_active (bool default true)
- created_at, updated_at

4) PriceLists
- id (PK)
- code (unique, indexed)
- name
- currency (default EGP)
- valid_from (nullable), valid_to (nullable)
- is_active (bool default true)
- notes (nullable)
- created_at, updated_at

5) PriceListItems
- id (PK)
- price_list_id (FK, indexed)
- service_id (FK, indexed)
- price (decimal(18,2), required)
- min_discount_pct (decimal(5,2) nullable)
- max_discount_pct (decimal(5,2) nullable)
- UNIQUE(price_list_id, service_id)
- created_at, updated_at

6) PriceAdjustmentsLog (audit trail)
- id (PK)
- price_list_id (FK)
- action_type (PCT / FIXED / SET)
- value (decimal(18,4))
- filter_department_id (nullable)
- filter_category (nullable)
- affected_count (int)
- created_by (nullable)
- created_at

RULES / VALIDATIONS
- Service.code must be unique and required
- Service.department_id required
- Accounts and cost center required on service
- PriceList.code unique and required
- PriceListItem.price must be >= 0
- Bulk adjustment must:
  - Support +% / -% and +fixed / -fixed
  - Optional filters: department, category, active only
  - Provide a preview: show affected_count + sample of before/after for first 20 rows
  - Confirm action
  - Apply changes in a single DB transaction
  - Round to 2 decimals using standard rounding
  - Write one row to PriceAdjustmentsLog
