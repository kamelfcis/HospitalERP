أريدك أن تقوم بمراجعة تصميمية دقيقة (Design Decision Review) لمنطق تسعير الأصناف (الأدوية) في النظام الحالي، لتحديد ما إذا كان يجب الاستمرار في الاعتماد على **Client-driven pricing** كما هو، أم التحول إلى **Server-enforced pricing عند الحفظ/الاعتماد**.

❗ مهم:
- لا تنفذ أي تغيير فعلي الآن.
- الهدف: تحليل + قرار معماري فقط.
- لا تكسر السلوك الحالي.
- لا تقترح حلول نظرية منفصلة عن الواقع الحالي للنظام.
- اعتمد على الكود الموجود فعليًا (items pricing flow + /api/pricing + save/finalize).

━━━━━━━━━━━━━━━━━━━━━━━
🎯 الوضع الحالي (كما هو مطبق)
- الواجهة تستدعي: GET /api/pricing
- يتم تحديد السعر (بما في ذلك سعر القسم) عند إضافة الصنف
- يتم حفظ unitPrice كما أرسله الـ UI
- السيرفر لا يعيد احتساب السعر عند CREATE / UPDATE / FINALIZE
- لا يوجد enforcement على server يمنع تغيير السعر يدويًا

━━━━━━━━━━━━━━━━━━━━━━━
🔍 المطلوب منك تحليله

### 1) تقييم الوضع الحالي (Client-driven)
قيّم هل الاعتماد الحالي على الـ UI في تحديد السعر:
- آمن بما يكفي؟
- مناسب لبيئة ERP إنتاجية؟
- معرض لأي:
  - request tampering
  - stale data
  - race conditions
  - bypass غير مقصود

❗ اربط التقييم بواقع النظام الحالي (ليس نظريًا فقط)

━━━━━━━━━━━━━━━━━━━━━━━
### 2) هل يوجد Risk فعلي؟
حدد بوضوح:
- هل يمكن فعليًا إدخال سعر خاطئ للصنف رغم وجود سعر قسم؟
- هل scope-guard أو أي validation حالي يمنع ذلك؟
- هل المستخدم يمكنه:
  - تعديل السعر يدويًا
  - أو إرسال request مختلف

❗ لو الخطر نظري فقط، اذكر ذلك
❗ لو عملي، وضّح السيناريو

━━━━━━━━━━━━━━━━━━━━━━━
### 3) تقييم خيار Server-Enforced Pricing
قيّم ماذا يحدث لو قررنا:

- إعادة احتساب سعر الصنف في السيرفر عند:
  - CREATE
  - UPDATE
  - FINALIZE

راجع:
- هل هذا سيتعارض مع:
  - manual override الحالي
  - مرونة المستخدم
  - workflows الحالية
- هل سيكسر backward compatibility؟
- هل سيحتاج audit إضافي؟
- هل سيؤثر على الأداء؟

━━━━━━━━━━━━━━━━━━━━━━━
### 4) اقتراح الخيارات الواقعية (بدون تنفيذ)
أريد منك اقتراح 2 أو 3 سيناريوهات واقعية فقط، مثل:

Option A — الإبقاء على الوضع الحالي:
- Client-driven + audit/logging

Option B — Server validation فقط:
- لا يغير السعر
- لكن يتحقق أن السعر ضمن حدود مقبولة

Option C — Server authoritative:
- يعيد حساب السعر ويكتب فوق client

❗ لا تبتكر حلول كثيرة — ركز على الخيارات الواقعية القابلة للتطبيق في النظام الحالي

━━━━━━━━━━━━━━━━━━━━━━━
### 5) توصيتك النهائية
اختر أفضل خيار للنظام الحالي واذكر:

- لماذا هو الأنسب
- ما الذي سيكسبه النظام
- ما الذي قد يخسره
- هل التنفيذ الآن ضروري أم يمكن تأجيله

━━━━━━━━━━━━━━━━━━━━━━━
📦 المطلوب في الرد النهائي

1) تقييم الوضع الحالي (Client-driven)
2) هل يوجد خطر فعلي أم لا
3) تحليل تأثير التحول إلى server-enforced
4) عرض 2–3 خيارات فقط
5) التوصية النهائية
6) هل يحتاج تنفيذ الآن أم لاحقًا
7) أي متطلبات مستقبلية (audit / permissions / flags)

━━━━━━━━━━━━━━━━━━━━━━━
⚠️ تنبيهات صارمة
- لا تنفذ أي كود
- لا تغيّر سلوك قائم
- لا تفترض وجود مشاكل غير مثبتة
- لا تخلط بين service pricing و item pricing
- ركّز فقط على الأدوية/الأصناف
- اربط كل تحليل بالكود الحالي وليس تخمينات

اشتغل كأنك بتتخذ قرار معماري في نظام ERP production فعلي.
