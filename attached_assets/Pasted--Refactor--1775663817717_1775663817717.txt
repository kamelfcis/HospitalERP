أريدك أن تنفّذ **Refactor كامل ومنظم جدًا** لثلاث مناطق رئيسية في النظام الحالي، مع كل ما يرتبط بها مباشرة، بحيث تصبح بسيطة، واضحة، قابلة للصيانة، وآمنة جدًا لأي مبرمج لاحق حتى لو كان مستواه ضعيفًا.

الشاشات المستهدفة:
1) شاشة فاتورة المريض
2) شاشة حالات دخول المستشفى / Admissions
3) شاشة الخدمات والأسعار / Services + Price Lists + related pricing UI

❗ مهم جدًا:
- لا أريد إعادة كتابة من الصفر.
- لا أريد هدم أي flow شغال.
- لا أريد تحسينات شكلية فقط.
- المطلوب Refactor حقيقي، معماري، منظم، وقابل للصيانة.
- لازم تحافظ على نفس السلوك الحالي Business-wise إلا إذا اكتشفت bug حقيقي واضح.
- أي تعديل لازم يكون incremental, safe, backward-compatible.
- اشتغل بعقلية ERP/HMIS production حقيقي، وليس مشروع تجريبي.

- الكود واضح جدًا
- الملفات قصيرة نسبيًا
- كل جزء له مسؤولية واحدة
- الـ business logic ليست داخل JSX
- الـ queries ليست مبعثرة
- الـ naming واضح جدًا
- الـ types واضحة

- أي تعديل لاحق يمكن عمله بسهولة وبأقل احتمال لكسر النظام
- الأداء قوي
- والفهارس مدروسة وقوية

## المرحلة 3 — Refactor Execution Rules
نفّذ refactor على مراحل صغيرة وآمنة جدًا.

### قواعد صارمة:
1) لا تغيّر السلوك الخارجي إلا لو bug
2) كل مرحلة قابلة للاختبار
3) لا تنقل logic لمكان جديد ثم تترك النسخة القديمة
4) لا تكرر نفس rule في أكثر من مكان
5) routes تظل thin
6) storage/services تظل واضحة
7) React components تظل presentational قدر الإمكان
8) hooks لا تجمع 10 مسؤوليات معًا
9) لا تستخدم any
10) لا تكتب comments كثيرة بديلة عن تصميم جيد — خلي التصميم نفسه واضح

━━━━━━━━━━━━━━━━━━━━━━━
## المرحلة 4 — Strong Componentization
أريد componentization محترم جدًا، لكن بدون مبالغة.

### المطلوب:
- أي table كبيرة تتقسم
- أي row معقدة تتفصل
- أي badge/summary متكرر يتفصل
- أي lookup له logic خاص يتفصل
- أي modal/form كبير يتقسم sections
- أي behavior متكرر يتجمع في hook/helper

### لكن:
- لا تفصل كل 5 أسطر في ملف مستقل
- لا تبالغ في abstraction
- اجعل التنظيم يخدم مبرمجًا عاديًا، لا يربكه

## المرحلة 6 — Routes / Storage / Services Cleanup
أريد تنظيف طبقة الخادم كذلك:

### المطلوب:
- routes تكون thin جدًا
- أي logic حقيقي يذهب إلى storage/service/helper
- لا تكرر نفس SQL في أكثر من مكان
- أي query مشتركة يتم توحيدها
- أي consolidation logic أو pricing logic أو summary logic يكون له مصدر واحد

### تأكد من:
- typed return shapes
- naming واضح للـ methods
- فصل queries التحليلية عن queries التشغيلية
- عدم خلط admission logic مع visit logic بشكل فوضوي

- أسماء functions
- أسماء hooks
- أسماء components
- أسماء helpers
- أسماء variables
- أسماء DTOs / response models

### الهدف:
أي مبرمج يفتح الملف يفهم:
- هذا الملف مسؤوليته ماذا
- هذه الدالة ترجع ماذا
- هذا component يعرض ماذا
- هذا hook يدير ماذا

❗ لا تغيّر أسماء عامة مشهورة في كل النظام إلا لو لازم، لكن نظّف ما يخص هذه المناطق بوضوح.

━━━━━━━━━━━━━━━━━━━━━━━
## المرحلة 9 — Bug Prevention / Safety
أريد أن تراعي refactor مع منع البَجّات التي قد تظهر عادة في هذا النوع من الشاشات:

### Patient Invoice
- stale state
- اختلاف السعر بين UI والحفظ
- duplicate line logic
- broken permission checks
- accidental re-renders
- N+1 في lookups
- lost source metadata
- null / undefined handling

### Admissions
- duplicate totals
- consolidated confusion
- visit grouping confusion
- leakage of cross-department financial details
- unstable ordering
- wrong latest invoice

### Services & Pricing
- wrong department filtering
- wrong default price list selection
- stale resolver behavior
- inconsistent contract fallback
- UI preview ≠ server result

❗ refactor يجب أن يقلل فرص هذه الأخطاء، لا أن ينقلها فقط من مكان لمكان.

━━━━━━━━━━━━━━━━━━━━━━━
## المرحلة 10 — Output Style أثناء التنفيذ
أريدك أن تعمل بهذا الأسلوب:

### أولًا:
اعرض:
1) Current state mapping
2) الملفات الحرجة
3) خطة refactor المقترحة
4) ترتيب التنفيذ
5) المخاطر
6) كيف ستتجنبها

### ثم:
ابدأ التنفيذ على دفعات منظمة
- Batch 1
- Batch 2
- Batch 3
بحيث كل batch يكون coherent وقابل للفهم

### وفي النهاية:
أريد تقريرًا نهائيًا منظمًا جدًا فيه:
1) ما الذي تم refactor له
2) ما الذي لم يُلمس intentionally
3) كيف صار structure أوضح
4) ما الـ helpers/components الجديدة
5) ما الـ hooks التي تم تبسيطها
6) ما الـ indexes التي أضيفت
7) ما bugs التي تم منعها
8) كيف ضمنت backward compatibility
9) أي نقاط مؤجلة intentionally
10) Done looks like
