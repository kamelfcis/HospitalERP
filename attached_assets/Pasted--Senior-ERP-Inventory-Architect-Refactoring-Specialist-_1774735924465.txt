أنت الآن Senior ERP Inventory Architect + Refactoring Specialist + Import/Export Workflow Engineer.

مطلوب تصميم وتنفيذ شاشة جديدة باسم:

Opening Stock / الرصيد الافتتاحي للمخزن

⚠️ مهم جداً:
هذه الشاشة يجب أن تُعتبر من شاشات الصيدلية وتظهر ضمن Pharmacy Mode، وليست شاشة عامة مبهمة أو وظيفة مخفية خارج قائمة الصيدلية.

لكن:
❌ لا تبدأ التنفيذ مباشرة
✅ ابدأ أولاً بتحليل الكود الحالي
✅ استخرج خطة تنفيذ مفصلة
✅ راعِ إعادة استخدام الموجود وعدم تكرار الكود
✅ بعد عرض الخطة توقف وانتظر الموافقة

========================================
الهدف الوظيفي
========================================

أريد شاشة عملية ومنظمة لإدخال الرصيد الافتتاحي لمخزن محدد، بحيث أستطيع:

1) اختيار المخزن
2) إدخال الأصناف الخاصة بالمخزن
3) إدخال الكمية
4) اختيار الوحدة من الوحدات المعرّفة للصنف
5) إدخال سعر الشراء الافتتاحي
6) إدخال سعر البيع الافتتاحي
7) إدخال تاريخ الصلاحية إن كان الصنف يتطلب ذلك
8) دعم إدخال batch/lot إذا كان منطق النظام الحالي يدعمه
9) استيراد البيانات من Excel
10) تصدير Template Excel ثم إعادة تحميله مثل شاشة الأصناف
11) اعتماد / ترحيل الرصيد الافتتاحي بشكل آمن إلى المخزون

========================================
المهم جداً
========================================

لا أريد تكرار الكود الموجود في:
- import/export logic
- excel template generation
- parsing / validation
- item selectors
- unit options
- inventory posting patterns

إذا كانت هناك مكونات أو hooks أو helpers موجودة في شاشة الأصناف أو أي شاشة استيراد مشابهة:
✅ أعد استخدامها
✅ أو استخرج منها shared module
❌ لا تنسخها copy/paste

========================================
متطلب مهم خاص بالصيدلية
========================================

هذه الشاشة يجب أن:

1) تُصنّف ضمن شاشات الصيدلية
2) تظهر في الـ sidebar الخاص بالصيدلية
3) تكون متاحة في Pharmacy Mode
4) لا تُعامل كـ hospital-only
5) تدخل ضمن صلاحيات الصيدلية / المخزون / الإدارة حسب منطق الصلاحيات الحالي
6) تكون في مكان منطقي داخل navigation الصيدلي (مثل المخزون أو الإعدادات التشغيلية للصيدلية)

افحص وضعها الصحيح داخل الـ sidebar والصلاحيات ولا تضفها عشوائياً.

========================================
المطلوب منك في المرحلة الأولى فقط
========================================

أولاً: افحص المشروع وحدد بدقة:

1) هل يوجد حالياً أي منطق قريب من:
- opening balances
- stock adjustments
- opening entries
- inventory initialization
- initial lots
- stock count posting
- receiving/posting flows

2) أين توجد شاشات مشابهة يمكن إعادة استخدام منطقها:
- شاشة الأصناف (Excel import/export)
- شاشات الاستلام
- الجرد
- التحويل
- أي شاشة فيها item line editor

3) ما هي الجداول الحالية ذات الصلة:
- items
- inventory_lots
- inventory_lot_movements
- warehouses / stores
- stock transactions
- prices
- batches
- expiry fields
- أي header/lines pattern قائم في المشروع

4) ما أفضل طريقة تصميم:
- هل نعمل Opening Stock Header + Lines؟
- أم نستخدم stock adjustment type؟
- أم opening document type مستقل؟
- وما الأفضل محاسبياً وتشغيلياً؟

5) كيف سنضمن أن:
- الرصيد الافتتاحي لا يُحمّل مرتين لنفس المخزن بالخطأ
- الكمية تتحول داخلياً بشكل صحيح حسب منطق الوحدات الجديد
- الأسعار الافتتاحية لا تفسد متوسط التكلفة
- الأصناف التي تتطلب expiry/batch يتم التحقق منها
- الأصناف غير المتتبعة بالباتش لا تُفرض عليها باتشات

6) ما أفضل طريقة لإعادة استخدام import/export الحالي من شاشة الأصناف بدون duplication؟

7) أين ستوضع الشاشة داخل الصيدلية:
- في أي route
- في أي section من الـ sidebar
- وما الصلاحية المناسبة لها
- وهل يجب أن تظهر فقط للـ owner/admin أو أيضاً لمسؤولي المخزن

========================================
المخرجات المطلوبة من المرحلة الأولى
========================================

أريد تقريراً بعنوان:

OPENING STOCK SCREEN IMPLEMENTATION PLAN

ويحتوي على:

A) Current architecture assessment
- ما الموجود حالياً في المشروع ويمكن البناء عليه
- ما الشاشات أو الخدمات أو الـ helpers القابلة لإعادة الاستخدام
- ما الذي يحتاج refactor قبل الإضافة

B) Recommended design
- أفضل تصميم للشاشة
- هل ستكون Header + Lines
- ما شكل دورة العمل: draft → validate → post
- ما شكل العلاقة مع المخزن واللوت والرصيد والحركات

C) Data model proposal
- ما الجداول الجديدة المطلوبة إن وجدت
- أو هل يمكن استخدام جداول موجودة مع document type واضح
- الحقول المطلوبة في header
- الحقول المطلوبة في lines

D) Posting logic
- كيف سيتم ترحيل الرصيد الافتتاحي فعلياً
- كيف تُنشأ inventory lots
- كيف تُنشأ inventory movements
- كيف يتأثر سعر الشراء الافتتاحي
- كيف يتأثر سعر البيع الافتتاحي
- ما القيود اللازمة لمنع الترحيل المكرر

E) Validation rules
أريد قواعد واضحة جداً، مثل:
- المخزن إلزامي
- الصنف إلزامي
- الكمية > 0
- الوحدة من الوحدات المعرفة للصنف فقط
- التحويل يتم على السيرفر
- إذا الصنف expiry/batch tracked → batch/expiry إلزامي
- منع تكرار نفس الصنف/الباتش بشكل يسبب ازدواج
- منع post إذا كانت هناك سطور invalid

F) Excel import/export reuse plan
- ما الذي سيتم إعادة استخدامه من شاشة الأصناف
- ما الذي يحتاج abstraction أو shared helper
- ما الذي يجب منعه من التكرار

G) Pharmacy placement
- أين ستظهر الشاشة داخل Navigation الصيدلية
- ما route الخاص بها
- ما permission المطلوبة
- وكيف ستتعامل مع Pharmacy Mode

H) Refactor opportunities
- shared excel service
- shared line editor
- shared item/unit picker
- shared posting validation
- أي duplication يجب التخلص منه قبل التنفيذ

I) Files to change
أريد قائمة دقيقة:
- file path
- لماذا سيتعدل
- نوع التعديل (feature / refactor / validation / route / sidebar / API)

J) Acceptance criteria
Checklist واضحة جداً للنجاح، مثل:
- إنشاء رصيد افتتاحي لمخزن محدد
- إدخال صنف بوحدة صحيحة
- إدخال سعر شراء وسعر بيع افتتاحي
- import من Excel يعمل
- export template يعمل
- post ينشئ الرصيد مرة واحدة فقط
- الشاشة تظهر في Pharmacy Mode
- الشاشة لا تكسر أي flow حالي للاستلام أو البيع أو الجرد

K) Test plan
اختبارات يدوية واضحة تشمل:
- major only
- major + medium
- major + minor
- أصناف بباتش/صلاحية
- أصناف بدون باتش
- import صالح
- import غير صالح
- duplicate prevention
- post ثم منع إعادة post

========================================
تعليمات تصميم صارمة
========================================

1) لا أريد تنفيذًا سريعًا سطحيًا
2) لا أريد duplication
3) لا أريد copy/paste من شاشة الأصناف
4) لا أريد منطق تحويل وحدات جديد منفصل — استخدم المنطق المركزي الحالي فقط
5) لا أريد منطق batch/expiry جديد منفصل إذا كان هناك منطق قائم يمكن البناء عليه
6) لا أريد route أو sidebar entry مضافة بشكل عشوائي

========================================
قيود مهمة جداً
========================================

- لا تبدأ بالكود الآن
- لا تنفذ migration أو schema changes قبل عرض الخطة
- لا تضف الشاشة للـ sidebar قبل اعتماد التصميم
- لا تفترض أي شيء بخصوص posting إلا بعد فحص flows الحالية
