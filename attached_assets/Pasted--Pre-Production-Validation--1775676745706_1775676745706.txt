نحن الآن في مرحلة ما قبل الاعتماد النهائي (Pre-Production Validation) لملف "ملف المريض + الفاتورة المجمعة".

❗ مهم جدًا:
- لا نريد أي refactor جديد الآن
- لا نريد features جديدة
- لا نريد تعديل معماري
- الهدف فقط: التأكد أن النظام صحيح 100% (أرقام + منطق + أداء + طباعة)

==================================================
الهدف من هذه المرحلة
==================================================

التأكد أن:
- كل الأرقام صحيحة
- كل التجميعات صحيحة
- كل المدفوعات محسوبة صح
- كل طرق العرض (Visit / Department / Classification / Detailed) متطابقة
- الطباعة مطابقة للشاشة
- الأداء ثابت
- لا يوجد تضليل للمستخدم

==================================================
أولًا: Data Integrity Validation (الأهم)
==================================================

اختبر النظام باستخدام بيانات حقيقية/واقعية، وقدم تقريرًا واضحًا.

### لكل مريض يتم اختباره:

1) اجمع:
- كل الفواتير الأصلية (patient_invoice_headers + lines)
- كل المدفوعات المرتبطة بها

2) تحقق من:

A. Total Consistency
- مجموع الفواتير الأصلية = إجمالي الفاتورة المجمعة
- مجموع by department = نفس الإجمالي
- مجموع by classification = نفس الإجمالي
- مجموع detailed lines = نفس الإجمالي

B. Payments Consistency
- مجموع المدفوعات = المعروض في شاشة المريض
- المدفوع + المتبقي = صافي الفاتورة
- لا يوجد double counting

C. Cross-View Consistency
- نفس الرقم يظهر في:
  - Summary cards
  - by visit
  - by department
  - by classification
  - detailed

أي اختلاف → سجله كـ BUG واضح مع السبب.

==================================================
ثانيًا: Visit Logic Validation
==================================================

راجع منطق الزيارات:

1) مريض داخلي:
- هل كل admission يظهر كـ Visit منفصلة؟
- هل التجميع by visit صحيح؟
- هل الفواتير مرتبطة بالزيارة الصحيحة؟

2) مريض خارجي:
- هل يتم grouping صحيح باستخدام visit_group_id؟
- هل العرض لا يخلط بين زيارات مختلفة؟

3) حالات مركبة:
- مريض له:
  - admission + outpatient visits
- هل الفصل واضح؟
- هل التجميع لا يخلط بينهم؟

==================================================
ثالثًا: Payments Validation
==================================================

اختبر سيناريوهات:

1) دفع على فاتورة واحدة
2) دفع على أكثر من فاتورة
3) دفع من أكثر من خزنة
4) دفع جزئي

تحقق من:
- عرض الخزنة صحيح
- ربط الدفع بالفاتورة صحيح
- عدم تكرار المدفوع في التجميع
- توزيع المدفوع في التقارير لا يسبب تضليل

==================================================
رابعًا: Large Data & Performance Testing
==================================================

اختبر مريض به:

- 2+ admissions
- 1+ outpatient visit
- 50–200 line items
- مدفوعات متعددة

تحقق من:

- زمن تحميل:
  - Patient file
  - Consolidated tab
  - Detailed lines

- هل يوجد lag عند:
  - تغيير التبويب
  - تغيير filters
  - الطباعة

- هل هناك unnecessary re-renders؟

- هل pagination يعمل فعليًا في invoice-lines endpoint؟

==================================================
خامسًا: Print Validation (مهم جدًا)
==================================================

اختبر الطباعة في الأربع أوضاع:

1) By Visit
2) By Department
3) By Classification
4) Detailed

لكل وضع تحقق من:

- اسم المريض + الكود ظاهرين
- التاريخ/الفترة واضحة
- العناوين واضحة
- المجاميع النهائية صحيحة
- لا يوجد اختلاف عن الشاشة
- لا يوجد قص في البيانات عند تعدد الصفحات
- التنسيق مفهوم للمحاسب/الإدارة

==================================================
سادسًا: Permissions & Access
==================================================

تحقق من:

- هل كل مستخدم يرى فقط ما يخصه؟
- هل هناك فصل بين:
  - طبي
  - مالي
- هل أي endpoint مكشوف بدون صلاحية؟

==================================================
سابعًا: Edge Cases
==================================================

اختبر:

1) مريض بدون فواتير
2) مريض بدون مدفوعات
3) مريض لديه:
   - فواتير ملغاة
   - فواتير draft
4) مريض لديه فقط outpatient
5) مريض لديه فقط inpatient
6) مريض تم دمجه (merged patient)

==================================================
ثامنًا: المطلوب في الرد
==================================================

أريد تقرير واضح جدًا يحتوي على:

1) Test Scenarios Executed
- قائمة بكل السيناريوهات التي تم اختبارها

2) Data Validation Results
- هل الأرقام متطابقة أم لا
- مع أمثلة فعلية

3) Performance Results
- أزمنة تقريبية
- أي مشاكل تم ملاحظتها

4) Print Validation Results
- هل الطباعة مطابقة أم لا
- مع ملاحظات

5) Bugs Found
- قائمة واضحة:
  - الوصف
  - السبب المحتمل
  - severity (High / Medium / Low)

6) Risk Assessment
- هل النظام جاهز Production أم لا؟
- ما المخاطر المتبقية؟

7) Final Verdict
- Ready / Not Ready
- مع أسباب واضحة
