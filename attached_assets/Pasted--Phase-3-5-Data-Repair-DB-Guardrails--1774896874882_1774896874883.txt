أريد منك تنفيذ Phase 3.5: Data Repair + DB Guardrails بدقة شديدة جدًا، وبدون أي تغييرات خارج النطاق.

الهدف:
1) إصلاح البيانات القديمة المتضررة بأمان
2) منع تكرار نفس مشاكل المخزون والقيود مستقبلًا على مستوى قاعدة البيانات
3) عدم تنفيذ أي إصلاح destructive تلقائي بدون فحص وتوثيق

قواعد صارمة:
- لا تغيّر UI
- لا تنفذ refactor واسع

- أي repair يجب أن يمر بثلاث مراحل:
  1) detection query
  2) impact explanation
  3) repair query منفصلة
- فرّق بوضوح بين:
  confirmed corruption
  repairable safely
  manual decision required

TASK-DR-01: Duplicate lots repair plan
==================================

بناءً على النتيجة الحالية، هناك 2 duplicate lots مؤكدة.

المطلوب:
- اعرض تقريرًا تفصيليًا عن الحالتين:
  - lot ids
  - item_id / item name
  - warehouse_id / warehouse name
  - expiry_month / expiry_year
  - qty_in_minor لكل lot
  - عدد inventory_lot_movements المرتبط بكل lot
  - هل أحد الـ lots qty=0 أم لا
  - أي lot هو الأحدث وأي lot هو الأنسب للاحتفاظ به
- لا تنفذ الدمج مباشرة
- أعطني repair script آمن خطوة بخطوة
- إذا كان نقل الحركات من lot إلى lot آخر قد يسبب كسر traceability، وضّح ذلك بوضوح

أريد أيضًا:
- query تحقق بعد الدمج
- rollback idea إن أمكن

==================================
TASK-DR-02: Add DB guardrail against future duplicate lots
==================================

بعد مراجعة حالة الـ duplicate lots الحالية:

- اقترح القيد/الـ unique index الصحيح لمنع تكرار نفس lot مستقبلًا على مستوى DB
- راعِ التعامل الصحيح مع NULL في expiry_month / expiry_year
- إذا كان unique index المباشر قد يفشل بسبب البيانات الحالية:
  - لا تطبقه فورًا
  - جهز migration conditional:
    - يفحص أولًا وجود duplicates
    - إذا لا يوجد duplicates → ينشئ index
    - إذا يوجد → يسجل warning واضح ولا ينشئه
- أريد الكود الفعلي المناسب في startup migration أو migration script

المهم:
أريد DB-level prevention، وليس اعتمادًا فقط على application logic.
