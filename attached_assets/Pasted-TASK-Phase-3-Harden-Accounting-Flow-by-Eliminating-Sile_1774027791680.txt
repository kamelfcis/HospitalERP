TASK: Phase 3 — Harden Accounting Flow by Eliminating Silent Failures, Tracking Journal Problems, and Making Collection Completion Recoverable

Phase 1 and Phase 2 are fully complete and closed.
Now harden the accounting flow itself.

BUSINESS / CONTROL GOAL
No accounting event should fail silently.
Any journal generation failure, completion failure, or accounting-side exception must become explicit, durable, queryable, and recoverable.

IMPORTANT PHASE SCOPE
Do NOT do a broad redesign yet.
Do NOT yet split the pharmacy sale journal and collection journal into two separate accounting models in this phase.
First, make the CURRENT design safe, traceable, and recoverable with minimal-risk refactor.

CONFIRMED CURRENT WEAK POINTS
- completeSalesJournalsWithCash runs after commit in a fire-and-forget path
- if it fails, the invoice may be collected while the journal is not fully completed
- doctor settlement journal generation can swallow errors
- patient invoice journal failures are not tracked durably
- some accounting paths can effectively skip or hide failure outcomes
- sales may generate entries with periodId = null if no open fiscal period exists
- retryFailedJournals exists for some failures, but collection completion failures are not durably tracked/retried

IMPLEMENTATION GOALS
1) Make all accounting failures explicit
2) Persist accounting event/failure status durably
3) Make collection-completion failures visible and recoverable
4) Remove swallowed accounting exceptions
5) Add a minimal admin visibility/reporting path for failed/pending accounting events
6) Preserve idempotency and current business flow as much as possible

TARGET DESIGN
Implement a durable accounting event tracking model.
Prefer reusing an existing focused table if it already fits.
If not, create a dedicated table such as accounting_event_log or journal_event_log.
